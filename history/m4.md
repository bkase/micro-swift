# Milestone 4 — Fallback Automaton Path and Custom Metal Kernels (Revised Runtime Spec)

Status as of March 6, 2026:

- Milestone 2 is implemented in `MicroSwiftLexerGen`, including `classSets`, stable token-kind IDs, `runtimeHints`, and row-major fallback payloads encoded in `RulePlan.fallback` inside `LexerArtifact`.
- The frontend paging/types from Milestone 1 exist in `MicroSwiftFrontend` as `PreparedSource`, `SourceTape`, `HostLineIndex`, `PagePolicy`, `SourcePage`, and `SourcePaging.planPages(...)`.
- Milestone 3 is implemented in `MicroSwiftTensorCore`, including `ArtifactRuntime`, `ArtifactLoader`, `PagingShell`, `TensorLexer.lexPage(...)`, `LexingShell`, `LexOptions`, `PageLexResult`, greedy selection, coverage/error emission, and keyword remap.
- The current runtime profile is still `v0`: `CapabilityValidator` rejects both `RuleFamily.localWindow` and `RuleFamily.fallback`, and `RuleBuckets` / `TensorLexer.lexPage(...)` only execute the fast-path families (`literal`, `runClassRun`, `runHeadTail`, `runPrefixed`) plus keyword remap and packed-row emission.
- The current `MicroSwift.v0` spec in `MicroSwiftLexerGen` still fits that fast-path runtime profile: whitespace, line comments, identifiers/keywords, integers, and punctuation all lower to `literal` or `run`.
- Milestone 4 extends the runtime to support `RuleFamily.fallback` while keeping `RuleFamily.localWindow` out of scope.

Unless a section explicitly says "current repo state", the code sketches below describe the intended Milestone 4 architecture rather than the exact current Swift signatures. The checked-in Milestone 3 implementation still uses host arrays in `TensorLexer.lexPage(...)` and `PageLexResult`, with CLI capability checks enforced before lexing.

## Objective

Implement the **second runtime lexer profile** for Micro-Swift by extending the Milestone 3 page-oriented tensor pipeline with deterministic execution of **artifact-emitted fallback automata** in MLX-Swift.

This milestone must preserve the Milestone 3 contract:

- **artifact-driven**, not regex-driven
- **deterministic**, byte-for-byte across runs
- **page-oriented**, using the frontend paging contract
- **shape-stable** in the warm path
- **span-precise** for lexical errors
- **honest about runtime coverage**

The main change relative to Milestone 3 is that the runtime may now execute:

- `RuleFamily.literal`
- `RuleFamily.run`
- `RuleFamily.fallback`
- keyword remaps from `keywordRemaps`

It still does **not** execute:

- `RuleFamily.localWindow`

Milestone 4 therefore does **not** reopen lexing semantics. It only adds one more artifact-defined rule family to the same canonical runtime pipeline.

---

## Scope

### In scope

Milestone 4 owns:

- the loader/runtime bridge for the artifact’s flattened fallback tables
- loader-time capability validation for the fallback-capable runtime profile
- a bounded **128-state** fallback executor
- a custom Metal kernel path for page-local fallback execution when stock MLX composition is awkward
- fallback candidate generation that integrates into the Milestone 3 winner-reduction pipeline
- deterministic handling of fallback/fast-family overlap
- constant-buffer placement and kernel reuse policy for fallback tables
- verification that fallback execution matches a scalar reference evaluator
- golden tests, property tests, differential tests, and fallback-heavy fixture artifacts
- lex-only benchmark hooks for fallback activity and warm-path cache reuse

### Explicitly out of scope

Milestone 4 does **not** own:

- `localWindow` runtime execution
- regex parsing at runtime
- NFA/DFA construction at runtime
- backtracking regex execution
- host scalar lexing fallback
- parsing, structural indexing, symbol resolution, typing, or codegen
- string literal support unless a future generated language profile requires it

If an artifact requires `RuleFamily.localWindow`, the loader must reject it for this runtime profile.

## Design constraints

### 1. Functional core, imperative shell

The runtime core stays aligned with Milestone 3:

```swift
lexPage(
  bytesPadded: MLXArray<UInt8>,   // shape [P]
  validLen: Int32,                // <= P
  baseOffset: Int64,
  artifact: ArtifactRuntime,
  options: LexOptions
) -> PageLexResult
```

The imperative shell owns:

- file IO
- `SourcePage` planning / consumption
- overflow checks
- runtime-profile capability validation
- compiled-graph caching and warmup
- custom Metal kernel creation and reuse
- concatenation of page results
- CLI / dump formatting

The functional core owns:

- byte classification
- ClassSet recovery
- per-rule candidate generation
- fallback family execution
- winner reduction
- greedy non-overlap selection
- keyword remap
- coverage computation
- lexical error-run construction
- packed row emission

### 2. Runtime semantics come only from the artifact

Milestone 4 consumes:

- `runtimeHints`
- `tokenKinds`
- `byteToClass`
- `classes`
- `classSets`
- `rules`
- `keywordRemaps`
- flattened fallback tables emitted by the Milestone 2 generator

It does **not** reconstruct semantics from:

- rule names
- regex source text
- ad hoc host-side regex engines
- hardcoded keyword logic
- assumptions like “digit is one byte class”

### 3. Milestone 4 preserves the Milestone 3 pipeline

The canonical page-local pipeline remains:

```text
bytes
-> classID
-> class-set membership masks / bitsets
-> per-rule candidate tensors / family-local winner tensors
-> hierarchical winner reduction
-> greedy non-overlap selector
-> keyword remap
-> coverage mask
-> lexical error runs
-> final keep/filter + compact packed rows
```

Milestone 4 changes **only** the candidate-generation layer by allowing the fallback family to contribute a candidate winner.

### 4. Fixed 128-state fallback representation

Milestone 4 uses a fixed-capacity **128-state** fallback representation.

There is no dynamic “choose 64 or 256 later” machinery in this runtime profile. The artifact/runtime bridge and kernel contract are built around:

- two 64-bit words for the active-state set
- two 64-bit words for step targets
- two 64-bit words for accept masks

If an artifact’s flattened fallback tables require more than 128 states, the loader must reject the artifact deterministically.

### 5. Shape stability matters

The warm path must keep MLX graphs and Metal pipelines reusable:

- fixed page buckets
- fixed tensor shapes per page bucket
- fixed packed-row capacity
- deterministic padding
- fixed fallback table shapes after loader-time packing

Dynamic graph structure inside the hot path is forbidden.

### 6. Correctness beats unsound vectorization

As in Milestone 3:

- fallback execution may use a custom kernel if stock MLX composition is awkward
- that kernel must still be pure, deterministic, and page-local
- it is acceptable to do a bounded linear loop in a device kernel when that is the clearest correct formulation

No optimization is normative unless it preserves the exact artifact-defined semantics.

## Loader-time runtime profile validation

Milestone 3 validated the fast-path profile. Milestone 4 introduces a second runtime profile:

### Runtime profile v1-fallback

Accepted rule families:

- `.literal`
- `.run`
- `.fallback`

Rejected rule families:

- `.localWindow`

### Required fallback validations

Before lexing, the loader must validate all fallback-related artifact constraints:

1. every fallback rule has a valid flattened fallback payload
2. the combined fallback state count is `<= 128`
3. the flattened tables have deterministic, internally consistent sizes
4. fallback widths are bounded
5. fallback widths do not exceed `runtimeHints.maxDeterministicLookaheadBytes`
6. rule metadata needed for winner comparison exists and is total:
   - `ruleID`
   - `priorityRank`
   - `tokenKindID`
   - `mode`

If validation fails, the runtime must refuse to lex and emit a deterministic capability diagnostic such as:

```text
artifact-capability-error:
unsupported fallback payload for runtime profile v1-fallback
ruleID=...
name=...
reason=state-cap-exceeded|missing-table|width-exceeded|max-lookahead-mismatch|localWindow-present
```

### Important consistency point with `MicroSwift.v0`

The current generated `MicroSwift.v0` artifact may still contain **no** fallback rules. That is fine.

In that case:

- the loader validates the artifact successfully under v1-fallback
- the fallback runtime tables are empty / disabled
- runtime behavior is identical to Milestone 3

This is important because Milestone 4 must be testable on dedicated fallback-heavy fixture artifacts even before the main Micro-Swift language profile needs fallback rules.

## Artifact runtime bridge

Milestone 4 extends the Milestone 3 `ArtifactRuntime` bridge rather than inventing a separate runtime entry point.

### Required runtime constants

Milestone 4 still requires everything Milestone 3 required:

- `maxLiteralLength`
- `maxBoundedRuleWidth`
- `maxDeterministicLookaheadBytes`
- `byteToClassLUT[256]`
- runtime `ClassSet` representation
- fast-family rule buckets
- keyword remap tables

It additionally requires MLX-friendly fallback constants derived from the artifact’s flattened fallback section:

- `fallbackEnabled`
- `fallbackNumStatesUsed`
- `fallbackMaxWidth`
- `fallbackStartMaskLo`
- `fallbackStartMaskHi`
- flattened step tables in a constant-buffer-friendly layout
- per-rule accept masks
- per-rule global metadata
- fallback start-class summaries used for quick eligibility checks

### Internal runtime representation

A reasonable internal shape is:

```swift
public struct ArtifactRuntime {
  public let runtimeHints: RuntimeHintsRuntime
  public let byteToClassLUT: MLXArray<UInt16>
  public let classSets: ClassSetRuntime
  public let fastBuckets: FastRuleBucketsRuntime
  public let keywordRemaps: KeywordRemapRuntime
  public let fallback: FallbackRuntime?
}
```

With:

```swift
public struct FallbackRuntime {
  public let numStatesUsed: UInt16        // <= 128
  public let maxWidth: UInt16
  public let startMaskLo: UInt64
  public let startMaskHi: UInt64

  public let stepLo: [UInt64]             // flattened [class, state]
  public let stepHi: [UInt64]             // flattened [class, state]

  public let acceptLoByRule: [UInt64]
  public let acceptHiByRule: [UInt64]

  public let globalRuleIDByFallbackRule: [UInt16]
  public let priorityRankByFallbackRule: [UInt16]
  public let tokenKindIDByFallbackRule: [UInt16]
  public let modeByFallbackRule: [UInt8]

  public let startClassMaskLo: UInt64
  public let startClassMaskHi: UInt64
}
```

This is an internal runtime shape, not a claim about the public serialized artifact schema.

### ClassSet recovery still comes first

Fallback execution must use the same `byteToClass` and `classSets` bridge as the fast families. It must not reinterpret raw bytes with a separate classification scheme.

### Fallback rule bucketing

Unlike `literal` and `run`, fallback rules do not need to be emitted as a large `[R_fallback, P]` tensor of individual candidates.

Because the fallback artifact is already a flattened combined automaton, the runtime may execute fallback as a **single family-local winner kernel** that produces, for each start position:

- `len`
- `priorityRank`
- `ruleID`
- `tokenKindID`
- `mode`

This is fully consistent with Milestone 3’s hierarchical winner reduction: fallback simply contributes one more bucket-local winner tensor.

## Paging contract

Milestone 4 inherits the paging contract from Milestone 3.

The repo already has a deterministic host-side planner in `MicroSwiftFrontend`:

- `PagePolicy(targetBytes: ...)`
- `SourcePaging.planPages(...)`
- `SourcePage.oversize`

Milestone 4 must consume that planner or an equivalent replacement with the same observable invariants:

- pages are contiguous and non-overlapping
- every page end is a line start or EOF
- oversize single-line pages are flagged deterministically
- diagnostics continue to use stable byte offsets / spans from `MicroSwiftFrontend`

### Fallback and page boundaries

Fallback execution is page-local. It must not read from neighboring pages.

For any page-local position `i`, fallback stepping may examine only bytes inside the current `bytesPadded[0..<P]` buffer. If a step would move beyond `validLen`, that start position simply cannot match further and the active-state set becomes empty.

This keeps the runtime honest:

- no cross-page scalar repair
- no hidden host-side continuation
- no cross-page device synchronization

### Page safety and bounded lookahead

Milestone 4 does **not** introduce a separate halo buffer contract. It stays consistent with Milestone 3’s `bytesPadded: [P]` page input.

The shell is responsible for ensuring:

- the chosen page/bucket configuration is valid for the artifact’s `maxDeterministicLookaheadBytes`
- oversize / unsupported pages fail deterministically
- no host scalar lexing fallback is used

### Overflow policy

As in Milestone 3, if the planner produces an oversize page that the runtime cannot execute safely, the shell must fail lexing for that file with a precise diagnostic such as:

```text
lex-page-overflow:
line exceeds maximum supported page bucket
```

This remains normative even after fallback is added.

### Required fallback boundary fixtures

Milestone 4 must include fixtures for:

- fallback match ending exactly at page end
- fallback prefix beginning near page end but failing because the page ends
- fallback candidate fully contained inside a page
- fallback/fast-family overlap near page boundaries
- oversize page overflow that still rejects deterministically

The logical token fields remain:

- `kind`
- `flags`
- `startByte`
- `endByte`
- `payloadA`
- `payloadB`

### Hot-path transport layout

Milestone 4 continues to use the packed page-local transport model from Milestone 3:

```swift
public struct PageLexResult {
```

## Runtime pipeline

## Phase A — Byte classification and ClassSet recovery

This phase is unchanged from Milestone 3.

### A1. Byte-to-class lookup

```text
classID = gather(byteToClassLUT, bytes)
```

Shape: `[P]`

### A2. ClassSet membership

Semantic masks must still be derived from `classSets`, not singleton class assumptions.

Examples:

```text
isDigit      = classSetContains(digitSetID, classID)
isWS         = classSetContains(wsSetID, classID)
isIdentStart = classSetContains(identStartSetID, classID)
isIdentCont  = classSetContains(identContSetID, classID)
isNewline    = classSetContains(newlineSetID, classID)
```

### A3. Validity mask

```text
validMask[i] = (i < validLen)
```

Every later phase, including fallback, must respect `validMask`.

---

## Phase B — Per-rule candidate generation

Milestone 4 extends Phase B by adding the fallback family. The fast families from Milestone 3 remain unchanged.

### B0. Candidate generation is still bucketed

The runtime should still evaluate rules in homogeneous buckets:

- literal buckets by byte length
- run buckets by `RunPlan` shape
- one fallback family-local bucket

Fast buckets may still produce candidate tensors such as `[R_bucket, P]`. The fallback bucket may directly produce one winner tuple per start position.

---

### B1. `literal` execution

Unchanged from Milestone 3.

---

### B2. `run` execution

Unchanged from Milestone 3.

Supported `RunPlan` shapes remain:

- `classRun`
- `headTail`
- `prefixed`

---

### B3. `fallback` execution

This is the new core of Milestone 4.

### B3.1. Logical contract

For every page-local start position `i`, the fallback executor computes the best fallback candidate beginning at `i`, if one exists.

The output at each position is a winner tuple carrying at least:

- `len`
- `priorityRank`
- `ruleID`
- `tokenKindID`
- `mode`

A zero length means “no fallback candidate starts here”.

### B3.2. Start eligibility

The runtime may cheaply reject impossible starts using the fallback start-class mask derived from the artifact.

Conceptually:

```text
canStartFallback[i] =
  validMask[i] &&
  classID[i] ∈ fallbackStartClasses
```

If `canStartFallback[i] == false`, the fallback winner at `i` is the zero-length sentinel.

This optimization is valid because it is derived entirely from artifact/runtime tables.

### B3.3. 128-state active-set representation

The active state set is represented as two 64-bit words:

- `activeLo`
- `activeHi`

The initial active set is the artifact-defined start mask:

- `startMaskLo`
- `startMaskHi`

### B3.4. Fixed-trip bounded loop

For each start position `i`, fallback stepping uses a bounded loop:

```text
for k in 0..<fallbackMaxWidth
```

The loop is fixed-trip for shape stability and predictable SIMD behavior.

Each iteration checks whether `i + k < validLen`. If not, the active set becomes empty for that lane.

This avoids out-of-bounds reads without introducing host repair logic.

### B3.5. Step transition

At step `k`, read the current class:

```text
cls = classID[i + k]
```

Then compute the next active set by unioning the precomputed targets for all active states under that class.

Conceptually:

```text
next(active, cls) =
  OR { step[cls, s] | s ∈ active }
```

The runtime implementation may enumerate active bits with trailing-zero iteration or another deterministic bitset walk.

### B3.6. Accept detection and winner update

After each valid step, the runtime tests the active set against the accept masks of fallback rules.

Any accepting rules at the current length `k + 1` are compared using the same total order used elsewhere:

1. longer match wins
2. if lengths tie, smaller `priorityRank` wins
3. if that still ties, smaller `ruleID` wins

This produces the family-local fallback winner at start position `i`.

### B3.7. Why a custom kernel is acceptable here

This family-local stepping loop is exactly the sort of bounded, page-local, deterministic kernel that is awkward to express as a large graph of stock tensor ops.

Using an MLXFast custom Metal kernel here is acceptable because:

- the semantics are still pure
- the loop bound is fixed
- state is page-local
- no host fallback is introduced
- the rest of the lexer pipeline remains artifact-driven and tensor-oriented

## Phase C — Hierarchical winner reduction

The Phase C contract from Milestone 3 remains the same.

### Lexicographic comparator

Tie-breaking order remains:

1. longer length wins
2. if lengths tie, smaller `priorityRank` wins
3. if that still ties, smaller `ruleID` wins

### How fallback participates

Fallback is treated as one more bucket-local winner source.

So the reduction becomes:

1. reduce each literal bucket to one winner tuple per position
2. reduce each run bucket to one winner tuple per position
3. compute the fallback family-local winner tuple per position
4. reduce those bucket winners together with the same pairwise tree merge

This keeps graph depth shallow and preserves the Milestone 3 reduction story.

### Important consistency point

Milestone 4 does **not** replace Phase C with a separate “fallback-first” or “fallback-only” resolution path. There is still one deterministic global winner order per start position.

---

## Phase D — Greedy non-overlap selection

Milestone 4 does not change the Milestone 3 selector.

The normative selector remains the deterministic page-local greedy scan:

```text
coveredUntil = 0
for i in 0..<validLen:
  if bestLen[i] > 0 && i >= coveredUntil:
    isStart[i] = true
    coveredUntil = i + bestLen[i]
  else:
    isStart[i] = false
```

This is still required because:

- the selected token stream must be correct under maximal-munch semantics
- unsound prefix-max-over-all-candidates shortcuts remain unsound even after fallback is added

### Required overlap fixtures

Milestone 4 must add mixed-family overlap cases, including:

- fast literal vs fallback rule overlap
- fast run vs fallback rule overlap
- later valid token after an internal rejected fallback start
- fallback candidates beginning inside an already accepted fast token
- fallback candidates that would win at a start position but must still be rejected by coverage due to an earlier accepted token

---

## Phase E — Keyword remap

Keyword remap is unchanged from Milestone 3.

It remains:

- artifact-table-driven
- keyed by the selected token’s base rule or token kind
- applied only after non-overlap selection
- forbidden from changing spans

Fallback support must not introduce any hardcoded keyword logic.

---

## Phase F — Coverage and lexical error spans

Coverage is still computed from **all selected tokens before skip filtering**.

This remains critical because skip tokens from any family, including future fallback skip rules, still cover bytes and must not create fake lexical errors.

### F1. Coverage mask

As in Milestone 3:

- build a delta array from selected token spans
- prefix-sum it
- derive `covered[i] = (runningCount > 0)`

### F2. Unknown bytes

```text
unknown[i] = validMask[i] && !covered[i]
```

### F3. Error runs

Build maximal runs of unknown bytes and emit exact spans.

Fallback support does not change this logic.

---

## Phase G — Final keep/filter and packed transport emission

Unchanged from Milestone 3.

### G1. Keep mask

After coverage is computed:

- keep all non-skip selected tokens
- optionally keep skip tokens in debug mode
- keep lexical error runs

### G2. Compact and pack

Use prefix counts to assign row IDs in source order, then write packed rows into the fixed-capacity transport buffer.

Fallback-origin tokens must be packed exactly the same way as fast-family tokens.

### G3. Deterministic padding

Rows beyond `rowCount` are still zeroed or set to a fixed sentinel.

---

## Custom Metal kernel contract

Milestone 4 is where a custom kernel becomes first-class.

### Kernel responsibility

The fallback kernel should do exactly one job:

- compute the fallback family-local winner tuple for each page-local start position

It should **not** do:

- fast-family execution
- final global winner reduction
- greedy non-overlap selection
- keyword remap
- coverage computation
- token packing

### Kernel inputs

At minimum:

- `classID[P]`
- `validLen`
- fallback transition tables
- fallback accept masks
- fallback rule metadata
- fixed constants:
  - `P`
  - `fallbackMaxWidth`

### Kernel outputs

At minimum:

- `fallbackLen[P]`
- `fallbackPriorityRank[P]`
- `fallbackRuleID[P]`
- `fallbackTokenKindID[P]`
- `fallbackMode[P]`

A packed winner-tuple output is also acceptable as long as it is deterministic and fixed-shape.

### Determinism requirements

The fallback kernel must be deterministic by construction:

- no atomics
- no inter-thread communication
- no order-dependent writes
- one output row owned by one lane/thread
- fixed loop bounds
- fixed table layout
- fixed tie-break order

### Memory placement

The fallback tables must be bound as **constant** read-only buffers, not generic mutable device memory.

That includes:

- step tables
- accept masks
- rule metadata

This is part of the baseline kernel contract, not an optional tuning trick.

### Threadgroup staging

Threadgroup/shared-memory staging of the full step table is **not** part of the normative Milestone 4 design.

It may be introduced later as a benchmark-driven optimization, but Milestone 4 correctness and acceptance must not depend on it.

## Kernel cache and warm-path reuse

Milestone 4 must make kernel reuse explicit.

### Cache key

The fallback evaluator must be cached by a stable key such as:

```text
(deviceID, artifactHash, pageBucket, inputDType)
```

If page bucket and packed output shape are the same, warm runs must reuse the same compiled resources.

### Cached resources

A cache entry may include:

- the MLX compiled wrapper function
- the MLXFast custom Metal kernel / pipeline
- uploaded constant buffers or equivalent immutable runtime resources

### Logging policy

On cache miss, emit one structured record containing:

- `traceID`
- `event = "fallback-kernel-cache-miss"`
- `artifactHash`
- `pageBucket`
- `deviceID`

On cache hit: stay quiet.

On kernel creation failure, emit a structured error with the same fields and the failure reason.

## Determinism policy

Milestone 4 inherits all Milestone 3 determinism requirements and adds fallback-specific ones.

That means:

- artifact loading is deterministic
- fallback table packing is deterministic
- state numbering is stable in the artifact
- bucket ordering is deterministic
- fallback start eligibility is deterministic
- fallback accept comparison uses a total comparator
- global winner reduction uses the same total comparator
- non-overlap selection is deterministic
- keyword remap order is deterministic
- coverage and error grouping are deterministic
- page splitting is deterministic
- overflow handling is deterministic
- packed row padding is deterministic
- kernel cache keys are deterministic

Any sorting must use an explicit total order. Prefer not to sort in the hot path.

## Verification strategy

Milestone 4 should be even more aggressively verified than Milestone 3 because it adds a custom kernel and a second runtime profile.

### Unit tests

At minimum:

- fallback loader validation tests
- fallback runtime-bridge packing tests
- 128-state-cap rejection tests
- constant-table shape / count invariant tests
- fallback start-mask tests
- fallback accept-mask comparison tests
- mixed fast/fallback reduction tests
- mixed fast/fallback overlap-selector tests
- overflow-policy tests

### Property tests

At minimum:

1. same bytes + same artifact => same packed rows

2. token rows are in strictly increasing source order

3. token spans do not overlap

4. every valid byte is covered by either:
   - a selected token before filtering, or
   - an error run

5. keyword remap does not change token span

6. fallback family winner agrees with scalar reference on seeded random short pages

7. overflow behavior is deterministic for the same file and bucket config

### Differential tests

Milestone 4 must include a slow scalar reference evaluator for the fallback-capable runtime profile.

For random short inputs and dedicated fallback-heavy artifacts, assert that:

- fallback winner tuples agree with the scalar reference
- fast-family candidates still agree with the Milestone 3 reference
- global winner reduction agrees with a scalar reference
- greedy selector agrees with scalar left-to-right maximal munch
- final token tape agrees between device runtime and scalar reference

### Golden tests

Check in snapshots for:

- token streams from fallback-heavy fixture artifacts
- page-boundary fixtures mixing fast and fallback rules
- overlap fixtures
- loader-capability diagnostics
- fallback bridge dumps where useful

### Current `MicroSwift.v0` regression test

Because `MicroSwift.v0` may still have no fallback rules, Milestone 4 must include a regression test showing that:

- lexing `MicroSwift.v0` under the v1-fallback runtime profile produces the same token streams as Milestone 3

That proves the extension is conservative when fallback is unused.

### Generator chaos fuzzer

The runtime milestone should still benefit from a generator-facing chaos fuzzer, because malformed flattened fallback tables are a real risk surface.

For random bounded fallback-heavy lexer specs:

- the generator should either emit a valid deterministic artifact, or
- reject deterministically with a stable diagnostic

The runtime must never be expected to repair malformed semantics at load time.

## Acceptance criteria

Milestone 4 passes only when all of the following are true:

1. The runtime still consumes the full Milestone 2 bridge contract:
   - `runtimeHints`
   - `byteToClass`
   - `classes`
   - `classSets`
   - `tokenKinds`
   - `rules`
   - `keywordRemaps`
   - flattened fallback tables

2. The loader accepts artifacts requiring:
   - `literal`
   - `run`
   - `fallback`

   and rejects artifacts requiring:
   - `localWindow`

3. Semantic predicates are still recovered through `ClassSets`, not singleton byte-class assumptions.

4. Fallback execution is artifact-driven and does not reconstruct semantics from regex source or rule names.

5. The fallback runtime enforces the fixed **128-state** cap and rejects artifacts that exceed it.

6. The fallback family computes a deterministic family-local winner tuple per start position and integrates into the same hierarchical winner-reduction strategy used by the rest of the runtime.

7. Global best-match selection still uses the deterministic lexicographic comparator:
   - longer length
   - smaller `priorityRank`
   - smaller `ruleID`

8. Non-overlap selection remains the Milestone 3 greedy selector and is correct on mixed fast/fallback overlap fixtures.

9. Keyword remap is still artifact-table-driven and does not change spans.

10. Coverage and error runs are computed before skip filtering, and lexical error spans remain exact.

11. The runtime shell uses the deterministic frontend paging contract (`SourcePaging` / `SourcePage`) or an equivalent replacement, and emits deterministic overflow diagnostics instead of host scalar lexing fallback.

12. Packed page transport rows are deterministically padded and can be unpacked into the canonical logical token model.

13. The fallback kernel matches a scalar reference evaluator on seeded randomized tests and dedicated fallback-heavy fixtures.

14. The current `MicroSwift.v0` artifact, when it contains no fallback rules, still lexes identically to Milestone 3 under the new runtime profile.

15. The lex-only benchmark reports cold vs warm behavior and demonstrates compiled-resource reuse in the warm path.

## What this milestone buys the rest of the compiler

A correct Milestone 4 means the lexer runtime is no longer “fast for the easy cases only.”

After this milestone, later phases can assume:

- the lexer remains artifact-driven even when the generator needs residual regular rules
- fallback semantics are executed deterministically rather than repaired on the host
- the token tape remains stable, ordered, and span-precise
- lexical errors remain exact even when fallback rules are present
- the runtime contract is still honest about what it does **not** support (`localWindow`)

That matters because the whole architecture depends on a clean separation:

- Milestone 2 freezes regex semantics into an artifact.
- Milestone 3 turns the fast families into a high-throughput runtime.
- Milestone 4 closes the main residual gap by making fallback execution part of that same runtime rather than an external exception.

That is the point where later milestones can treat lexing as a solved, deterministic execution problem rather than a moving language-design target.
