# Milestone 3 — Tensor Lexer Fast Paths (Revised Runtime Spec)

Status as of March 6, 2026:

- Milestone 2 is implemented in `MicroSwiftLexerGen`, including `classSets`, stable token-kind IDs, flattened fallback tables, and `runtimeHints`.
- The frontend paging/types from Milestone 1 exist in `MicroSwiftFrontend` as `PreparedSource`, `SourceTape`, `HostLineIndex`, `PagePolicy`, `SourcePage`, and `SourcePaging.planPages(...)`.
- `MicroSwiftTensorCore` is still a stub, so this document is a target-state runtime spec, not a description of code that already exists.
- There is not yet a checked-in `Artifacts/MicroSwift.v0.lexer.json`; the current artifact surface is the `LexerArtifact` produced by `micro-swift lexergen dump`.
- The current generated `MicroSwift.v0` artifact already fits the intended Milestone 3 profile: all top-level rules are `literal` or `run`, and keyword handling is expressed only through `keywordRemaps`.

## Objective

Implement the **first runtime lexer profile** for Micro-Swift by executing the **Milestone 2 lexer artifact** as a **deterministic, page-oriented tensor pipeline** in MLX-Swift.

This milestone is the hot path for lexing. It must be:

- **artifact-driven**, not regex-driven
- **deterministic**, byte-for-byte across runs
- **shape-stable** in the warm path
- **span-precise** for lexical errors
- **honest about runtime coverage**

For the current generated `MicroSwift.v0` profile, Milestone 3 executes only:

- `RuleFamily.literal`
- `RuleFamily.run`
- keyword remaps from `keywordRemaps`

It does **not** execute:

- `RuleFamily.localWindow`
- `RuleFamily.fallback`

Those remain for Milestone 4 and later.

---

## Scope

### In scope

Milestone 3 owns:

- the runtime artifact bridge from Milestone 2 into MLX-friendly tables
- byte classification via `byteToClass`
- semantic predicate recovery via `classSets`
- execution of `LiteralPlan`
- execution of `RunPlan`:
  - `classRun`
  - `headTail`
  - `prefixed`

- deterministic best-candidate selection per start position
- deterministic greedy non-overlap selection
- keyword remap driven by artifact tables
- skip filtering driven by artifact `mode`
- lexical error-span production for unmatched bytes
- deterministic paging for giant files
- a hard overflow policy for overlong unsafe pages
- golden tests, property tests, and differential tests
- lex-only benchmark hooks

### Explicitly out of scope

Milestone 3 does **not** own:

- `localWindow` runtime execution
- `fallback` runtime execution
- custom DFA stepping for residual regexes
- parsing, structural indexing, symbol resolution, typing, or codegen
- string literal support unless the generated language spec is expanded to require it

If `MicroSwift.v0` does not include string literals, they are out of scope here and should surface as lexical errors.

## Design constraints

### 1. Functional core, imperative shell

The runtime core is a pure function:

```swift
lexPage(
  bytesPadded: MLXArray<UInt8>,   // shape [P]
  validLen: Int32,                // <= P
  baseOffset: Int64,
  artifact: ArtifactRuntime,
  options: LexOptions
) -> PageLexResult
```

The imperative shell owns:

- file IO
- page splitting
- overflow checks
- compiled-graph caching and warmup
- concatenation of page results
- CLI / dump formatting

No IO, no clocks, no randomness, and no hidden host-state dependence inside the tensor core.

### 2. Runtime semantics come only from the artifact

Milestone 3 consumes:

- `tokenKinds`
- `byteToClass`
- `classes`
- `classSets`
- `rules`
- `keywordRemaps`

It does **not** reconstruct semantics from:

- rule names
- regex source text
- hardcoded notions like “DIGIT is one class”
- implicit keyword special cases

### 3. One-way data flow

The page-local runtime pipeline is:

```text
bytes
-> classID
-> class-set membership masks / bitsets
-> per-rule candidate tensors
-> hierarchical winner reduction
-> greedy non-overlap selector
-> keyword remap
-> coverage mask
-> lexical error runs
-> final keep/filter + compact transport rows
```

Each stage derives new values from the previous stage. No in-place mutation of semantic state.

### 4. Shape stability matters

The warm steady-state path must keep MLX graphs reusable:

- fixed page buckets
- fixed batched shapes
- fixed output transport sizes
- deterministic padding of unused slots

Dynamic slicing inside the MLX graph is forbidden in the hot path.

### 5. Correctness beats clever but unsound vectorization

A vectorized formulation is only acceptable if it is correct for the current runtime profile.

In particular:

- raw prefix-max suppression over all candidate ends is **not normative**, because it is not correct for all overlap patterns in this profile
- any “optimization” that depends on metadata not present in the Milestone 2 artifact is not part of the contract

## Dependencies and runtime contract

### Depends on Milestone 1

Milestone 3 should build on the frontend surfaces already present in the repo:

- `PreparedSource`
- `SourceTape.bytes`
- `HostLineIndex`
- `PagePolicy`
- `SourcePage`
- `SourcePaging.planPages(...)`
- stable `ByteOffset` / `Span` utilities for diagnostics

### Depends on Milestone 2

The runtime bridge must consume the generated `LexerArtifact` fields explicitly:

- `runtimeHints`
- `tokenKinds`
- `byteToClass`
- `classes`
- `classSets`
- `rules`
- `keywordRemaps`

In the current codebase, the relevant `runtimeHints` contract is:

- `maxLiteralLength`
- `maxBoundedRuleWidth`
- `maxDeterministicLookaheadBytes`

Milestone 3 should treat those as loader-time constants rather than re-deriving page-boundary assumptions from regex source.

For the generated `MicroSwift.v0` artifact today, all three values are `2`, which is the concrete bound currently available to page-boundary logic.

### Loader-time runtime profile validation

Milestone 3 must validate the artifact **before lexing**.

For the first runtime profile:

- all `rules.family` must be either `.literal` or `.run`
- keyword remaps must be ordinary `keywordRemaps`
- no `localWindow` rules
- no `fallback` rules

If validation fails, the runtime emits a deterministic artifact-capability diagnostic and refuses to lex:

```text
artifact-capability-error:
unsupported rule family for runtime profile v0
ruleID=...
name=...
family=localWindow|fallback
```

This guardrail should almost never fire for the generated `MicroSwift.v0` artifact, because Milestone 2 already guarantees the current language profile stays within the first runtime profile.

## Artifact runtime bridge

The runtime bridge transforms the static artifact into MLX-friendly constants.

### Required runtime constants

At minimum:

- runtime-hint constants:
  - `maxLiteralLength`
  - `maxBoundedRuleWidth`
  - `maxDeterministicLookaheadBytes`
- `byteToClassLUT[256]`
- `classSet` membership representation
- rule tables:
  - `ruleID`
  - `tokenKindID`
  - `mode`
  - `priorityRank`
  - `family`
  - `firstClassSetID`
  - plan payload

- keyword remap tables
- rule buckets grouped by family / shape for efficient execution

### ClassSet representation

Semantic predicates like “digit”, “identifier-start”, “identifier-continue”, “whitespace”, and “newline” are **ClassSets**, not single byte classes.

The runtime bridge may choose either of these equivalent representations:

#### Dense mask form

```text
classSetMask[numClassSets, numByteClasses] : Bool/UInt8
```

Membership:

```text
classSetContains(setID, classID) =
  gather(classSetMask[setID], classID)
```

#### Bit-packed form

If `numByteClasses` is small enough, the bridge may instead pack each ClassSet into bit lanes:

```text
classSetBits[numClassSets, words]
```

with membership via lane gather + bit test.

The logical contract is the same:

```text
byteToClass[b] ∈ classSet(S)  iff  b ∈ S
```

The artifact does not care which runtime encoding is chosen.

### Rule bucketing for graph shape control

To avoid a long, deeply nested per-rule computation graph, the runtime bridge should pre-bucket rules into homogeneous groups, for example:

- literals by byte length
- `classRun` rules
- `headTail` rules
- `prefixed` rules by prefix length / stop-presence profile

Each bucket should have:

- fixed-shape payload tables
- a predictable candidate evaluation kernel
- a bounded per-bucket reduction shape

This keeps the compiled graph wide and shallow instead of long and scalar-looking.

## Canonical logical token model vs hot-path transport

### Canonical logical token model

The logical token fields remain:

- `kind`
- `flags`
- `startByte`
- `endByte`
- `payloadA`
- `payloadB`

That is the format downstream milestones reason about.

### Hot-path transport layout

Lexing is primarily memory-bandwidth bound. Writing six separate `[P]` arrays inside the hot path is expensive.

Therefore Milestone 3 should use a **packed page-local transport representation** for accepted tokens, then unpack into canonical logical form after page concatenation.

A recommended transport shape is:

```swift
public struct PageLexResult {
  public let packedRows: MLXArray<UInt64>   // or multiword packed rows
  public let rowCount: Int32
}
```

The exact bit layout is a runtime-profile constant, not part of the language artifact. It must be:

- deterministic
- sufficient for the largest supported page bucket
- zero-padded in unused slots

One acceptable profile packs:

- local start delta
- length
- token kind
- small flags

into one or two machine words.

If the chosen page buckets exceed the bit budget of a one-word layout, the runtime must use a multiword packed row rather than silently truncating.

### Deterministic padding requirement

Unused rows `[rowCount..<capacity]` must be zeroed or set to a sentinel. No uninitialized tail data is permitted.

## Paging contract

Paging is part of correctness, not an implementation detail.

The repo already has a deterministic host-side planner in `MicroSwiftFrontend`:

- `PagePolicy(targetBytes: ...)`
- `SourcePaging.planPages(...)`
- `SourcePage.oversize`

Milestone 3 should either consume that planner directly or replace it with an equivalent implementation that preserves the same observable invariants:

- pages are contiguous and non-overlapping
- every page end is a line start or EOF
- oversize single-line pages are flagged deterministically
- diagnostics continue to use stable byte offsets / spans from `MicroSwiftFrontend`

### Goals

- stable shapes in the common case
- safe rule execution near boundaries
- no hidden host scalar lexing fallback
- deterministic error behavior for pathological lines

### Safe-cut policy

The current host-side planner already cuts pages deterministically at newline-aligned boundaries or EOF using `PagePolicy(targetBytes:)`.

Milestone 3 may keep that policy as-is, or layer a small fixed runtime bucket set on top of it, but either way the runtime must not reinterpret source text to discover new cut points. The artifact/runtime contract for page boundaries should come from:

- frontend-provided `SourcePage` boundaries
- `runtimeHints.maxDeterministicLookaheadBytes`
- deterministic overflow handling for any page the runtime cannot execute safely

### Why newline alignment is not enough by itself

There are two separate concerns:

- multi-byte literals at the end of the page
- long runs and comments on long lines

Newline alignment solves cross-line issues, but it does **not** by itself solve overlong single-line inputs.

### Overflow policy for overlong lines

If the planner produces an oversize page that the runtime cannot execute safely, the shell must **not** fall back to host scalar lexing.

Instead, it emits a deterministic overflow diagnostic such as:

```text
lex-page-overflow:
line exceeds maximum supported page bucket
```

This may optionally surface as a synthetic CLI/debug token kind, but the normative behavior is:

- fail lexing for that file with a precise diagnostic
- include the source span of the overlong line if available
- do not stall the normal GPU path with CPU lexing

This keeps the runtime contract honest and avoids silent host/device synchronization cliffs.

### Required page-boundary fixtures

Milestone 3 must include fixtures for:

- `==` at page edges
- `->` at page edges
- identifier ending exactly at a page boundary
- integer ending exactly at a page boundary
- whitespace spanning a newline cut
- line comment ending just before a cut
- long line comment requiring a larger bucket
- overlong line exceeding the largest bucket and producing overflow

## Runtime pipeline

## Phase A — Byte classification and ClassSet recovery

### A1. Byte-to-class lookup

```text
classID = gather(byteToClassLUT, bytes)
```

Shape: `[P]`

### A2. ClassSet membership

Semantic masks must be derived from ClassSets, not assumed singleton classes.

Examples:

```text
isDigit      = classSetContains(digitSetID, classID)
isWS         = classSetContains(wsSetID, classID)
isIdentStart = classSetContains(identStartSetID, classID)
isIdentCont  = classSetContains(identContSetID, classID)
isNewline    = classSetContains(newlineSetID, classID)
```

Shape of each: `[P]`

### A3. Validity mask

```text
validMask[i] = (i < validLen)
```

Every phase must respect `validMask`.

### A4. Optional page-local newline helpers

For `prefixed(..., stopClassSetID: S)` plans, the runtime may precompute a page-local “next stop” helper using a reverse prefix-min or equivalent page-local structure. In the current `MicroSwift.v0` artifact this is a newline helper for line comments, but the runtime should stay generic over the stop ClassSet. This is valid because it depends only on page bytes.

---

## Phase B — Per-rule candidate generation

Milestone 3 executes the artifact’s plan shapes exactly.

Each rule produces, over the page:

- `candLen[rule, i]`
- optional `candFlags[rule, i]`

A zero length means “no start here”.

### B0. Candidate generation is bucketed, not scalar-looped

The runtime should evaluate rules in homogeneous buckets, not as a long host-side or graph-side chain of sequential updates.

Each bucket yields candidate tensors with shape like:

```text
[R_bucket, P]
```

or another fixed batch-friendly layout, then reduces them locally before participating in the global winner reduction.

This avoids linear graph-depth growth with rule count.

---

### B1. `literal` execution

For a literal plan with bytes `[b0, b1, ..., bL-1]`:

```text
startMask[i] =
  validMask[i..i+L-1] &&
  (bytes[i+0] == b0) &&
  ...
  (bytes[i+L-1] == bL-1)
```

Then:

```text
candLen[i] = L if startMask[i] else 0
```

Literal buckets should be grouped by literal length to maximize reuse of shifted comparisons.

---

### B2. `run` execution

Milestone 3 supports all three `RunPlan` shapes.

#### B2.1. `classRun(bodyClassSetID, minLength)`

Example: integer runs, whitespace runs.

Let:

```text
inBody = validMask && classSetContains(bodyClassSetID, classID)
```

Start mask:

```text
startMask[i] = inBody[i] && (i == 0 || !inBody[i-1])
```

End mask:

```text
endMask[i] = inBody[i] && (i == validLen-1 || !inBody[i+1])
```

Compact starts and ends in source order, pair them, compute lengths, then zero any start whose length is below `minLength`.

This is correct for maximal runs and deterministic because compaction order is deterministic.

#### B2.2. `headTail(headClassSetID, tailClassSetID)`

Example: identifiers.

Let:

```text
isHead = validMask && classSetContains(headClassSetID, classID)
isTail = validMask && classSetContains(tailClassSetID, classID)
```

Starts are bytes that begin a tail-run and satisfy head membership:

```text
startMask[i] = isHead[i] && (i == 0 || !isTail[i-1])
```

The run then extends through the maximal contiguous `isTail` segment.

As with `classRun`, pair starts to maximal tail-run ends deterministically.

#### B2.3. `prefixed(prefix, bodyClassSetID, stopClassSetID?)`

Example: line comments.

Prefix start:

```text
prefixStart[i] =
  validMask[i..i+L-1] &&
  bytes[i+k] == prefix[k] for all k
```

Then consume body bytes until the first byte whose class belongs to `stopClassSetID` if one exists, otherwise until the maximal valid body extent.

For the current `MicroSwift.v0` line-comment rule, this is:

- prefix: `//`
- body: `notNewline`
- stop: `newline`

The runtime should compute the first stop at or after `i + prefixLen` using page-local data only. No global line metadata is required inside the tensor core.

## Phase C — Hierarchical winner reduction

This phase chooses the best candidate **at each start position**.

### Lexicographic comparator

Tie-breaking order is:

1. longer length wins
2. if lengths tie, smaller `priorityRank` wins
3. if that still ties, smaller `ruleID` wins

This is a total deterministic order.

### Why a scalar sequential update is not normative

A long per-rule chain of:

```text
best = select(isBetter, cand, best)
```

builds a deep graph that compilers like MLX may struggle to optimize.

Milestone 3 should therefore use a **hierarchical reduction**:

1. evaluate candidates for a bucket in parallel
2. reduce within the bucket by pairwise tree merge
3. produce one winner tuple per bucket
4. reduce bucket winners by another pairwise tree merge

This gives graph depth on the order of:

```text
O(log R_bucket + log bucketCount)
```

instead of `O(totalRuleCount)`.

### Winner tuple

Each candidate winner should carry at least:

- `len`
- `priorityRank`
- `ruleID`
- `tokenKindID`
- `mode`

The reduction merges tuples using the lexicographic comparator above.

---

## Phase D — Greedy non-overlap selection

This is the stage that turns “best start at every byte” into a real tokenization.

### Why raw prefix-max suppression is not normative

A naïve formulation like:

```text
endAt[i] = i + bestLen[i]
prevMaxEnd = exclusivePrefixMax(endAt)
isStart[i] = i >= prevMaxEnd[i]
```

is **not correct in general** for this runtime profile.

Counterexample patterns like:

- `===` with rules `==` and `=`

can produce rejected internal starts whose end positions extend beyond the accepted token, which makes a raw prefix-max over all candidates unsound.

### Normative strategy: deterministic page-local greedy selector kernel

Milestone 3 therefore specifies a small, pure, page-local selection kernel with semantics:

```text
coveredUntil = 0
for i in 0..<validLen:
  if bestLen[i] > 0 && i >= coveredUntil:
    isStart[i] = true
    coveredUntil = i + bestLen[i]
  else:
    isStart[i] = false
```

This is:

- correct for left-to-right maximal-munch under the winner ordering from Phase C
- deterministic
- linear-time
- pure
- still device-resident

It is acceptable for this kernel to be expressed as a small generic TensorCore kernel if stock MLX composition is awkward. This is not “fallback regex execution”; it is infrastructure required for correctness.

### Why this is acceptable

Lexing is overwhelmingly memory-bound. A single linear page scan inside a device kernel is a better trade than an unsound or graph-hostile pseudo-vectorization.

### Required overlap fixtures

The test suite must include:

- `===`
- `===>`
- `---->`
- `/==`
- `////x`
- overlap cases where a longer low-priority candidate begins inside an accepted token
- overlap cases where a later valid token must still be accepted after an internal rejected candidate

These are the cases most likely to expose an invalid selection formulation.

---

## Phase E — Keyword remap

Keyword remap is driven strictly by the artifact’s `keywordRemaps`.

It is not hardcoded for “the identifier rule”.

For each selected token:

- inspect its `ruleID` or base token kind
- find remap tables keyed to that base rule
- compare the selected byte slice against remap entries
- replace the token kind on exact match

### Runtime strategy

For each remap table:

- use the table’s `maxKeywordLength`
- gather fixed windows from selected token starts
- compare against the table’s canonical entries
- apply the remapped kind if matched

If multiple remap tables exist, apply them in deterministic artifact order. In practice, the generated `MicroSwift.v0` artifact currently has one identifier-based table, but the runtime contract should not assume that.

---

## Phase F — Coverage and lexical error spans

Coverage must be computed from **all selected tokens before skip filtering**, otherwise whitespace and comments would appear as unknown bytes.

### F1. Coverage mask

Given selected token spans `[s, e)` in page-local coordinates, build:

- a delta array
- prefix-sum it
- derive `covered[i] = (runningCount > 0)`

This second scan pass is acceptable. It is shape-stable, linear, and much easier to reason about than clever local-window alternatives.

### F2. Unknown bytes

```text
unknown[i] = validMask[i] && !covered[i]
```

### F3. Error runs

Build maximal runs of `unknown`:

```text
errStart[i] = unknown[i] && (i==0 || !unknown[i-1])
errEnd[i]   = unknown[i] && (i==validLen-1 || !unknown[i+1])
```

Pair starts and ends and emit exact byte spans.

Error grouping must be deterministic.

---

## Phase G — Final keep/filter and packed transport emission

This stage produces the page result returned from `lexPage`.

### G1. Keep mask

After coverage is computed:

- keep all non-skip selected tokens
- optionally keep skip tokens in debug mode
- keep lexical error runs

This yields the final public token rows for the page.

### G2. Compact and pack

Use prefix counts to assign row IDs in source order, then write packed rows into a fixed-capacity transport array.

The transport must encode enough information to reconstruct:

- start byte
- end byte or length
- token kind
- minimal flags if needed

For Milestone 3, `payloadA` and `payloadB` can remain zero unless explicitly needed for diagnostics/debugging.

### G3. Deterministic padding

Rows beyond `rowCount` are zeroed or set to a fixed sentinel.

No stale device memory may leak into the transport.

## Determinism policy

Milestone 3 must be deterministic across repeated runs.

That means:

- artifact loading is deterministic
- rule bucketing order is deterministic
- candidate evaluation order within buckets is deterministic
- winner reduction uses a total comparator
- non-overlap selection is deterministic
- keyword remap table order is deterministic
- coverage and error grouping are deterministic
- page splitting is deterministic
- overflow handling is deterministic
- packed row padding is deterministic

Any sorting must use an explicit total order. Prefer not to sort at all in the hot path.

### Required structured failure record

On lex failure or invariant failure, emit a structured record with:

- trace ID
- file ID
- page bucket
- base offset
- valid length
- rule bucket counts
- selected token count
- error-run count
- overflow status if applicable

### On success

Success logging should stay quiet and structured. No chatty human logs in the hot path.

## Verification strategy

This milestone should be heavily verified because the core is pure and cheap to test.

### Unit tests

At minimum:

- byte-to-class runtime bridge tests
- class-set membership tests
- literal bucket tests
- `classRun` tests
- `headTail` tests
- `prefixed` tests
- keyword remap tests
- overlap selector tests
- overflow-policy tests

### Property tests

At minimum:

1. same bytes + same artifact => same packed rows
2. token rows are in strictly increasing source order
3. token spans do not overlap
4. every valid byte is covered by either:
   - a selected token before filtering, or
   - an error run

5. keyword remap does not change token span
6. overflow behavior is deterministic for the same file and bucket config

### Differential tests

Use a slow reference evaluator for the supported runtime profile:

- literal matching
- `classRun`
- `headTail`
- `prefixed`

For random short inputs, assert that:

- runtime candidate lengths agree with the slow evaluator
- winner reduction agrees with a scalar reference
- greedy selector agrees with a scalar left-to-right reference tokenization

### Golden tests

Check in snapshots for:

- token streams
- selected page-boundary fixtures
- overlap fixtures
- debug dumps of class sets and runtime rule buckets where useful

## Acceptance criteria

Milestone 3 passes only when all of the following are true:

1. The runtime consumes the full Milestone 2 bridge contract:
   - `runtimeHints`
   - `byteToClass`
   - `classes`
   - `classSets`
   - `tokenKinds`
   - `rules`
   - `keywordRemaps`

2. The loader rejects artifacts that require `localWindow` or `fallback`.

3. Semantic predicates are recovered through ClassSets, not singleton byte-class assumptions.

4. The runtime executes the actual `RunPlan` shapes from the artifact:
   - `classRun`
   - `headTail`
   - `prefixed`

5. Per-position best-match selection uses a deterministic lexicographic comparator.

6. Best-match reduction is implemented as a hierarchical bucket/tree reduction, not a long sequential scalar-style select chain.

7. Non-overlap selection is correct on overlap fixtures and is not based on the unsound raw prefix-max-over-all-candidates shortcut.

8. Keyword remap is artifact-table-driven and keyed by base rule, not hardcoded for one identifier rule.

9. Coverage and error runs are computed before skip filtering, and lexical error spans are exact.

10. The page splitter/runtime shell uses the deterministic frontend paging contract (`SourcePaging` / `SourcePage`) or an equivalent replacement, and emits deterministic overflow diagnostics instead of host scalar lexing fallback.

11. Packed page transport rows are deterministically padded and can be unpacked into the canonical logical token model.

12. Golden token streams are stable across repeated runs and process restarts.

13. The lex-only benchmark reports cold vs warm behavior and shows graph reuse in the warm path.

## What this milestone buys the rest of the compiler

A correct Milestone 3 gives later milestones a stable, deterministic token tape built from a frozen artifact contract.

That means Milestones 5 and beyond can assume:

- token boundaries are exact
- keyword kinds are already remapped
- lexical errors are already localized to precise byte spans
- the lexer is no longer “smart” at runtime; it is just an optimized executor of artifact plans

That separation is the whole point. Milestone 2 freezes regex semantics into a stable artifact. Milestone 3 turns that artifact into a high-throughput execution engine without re-opening language-design ambiguity.
