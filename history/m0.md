## Milestone 0 вЂ” Repository, toolchain, runtime, local verification, and Point-Free CLI foundation

**Goal:** establish a Swift-only repo that can build, test, lint, format, and run MLX-Swift command-line programs deterministically on a local machine, with one canonical verification path and a pre-commit hook that enforces it.

This milestone is no longer framed around CI. It is framed around a fresh clone being able to prove correctness locally.

The shell architecture for this milestone is intentionally Point-Free-shaped. `pfw` uses an `AsyncParsableCommand` root, depends on `swift-argument-parser`, `swift-dependencies`, `swift-custom-dump`, and inline snapshot testing, and tests commands by parsing the root command directly with `parseAsRoot(arguments)`, running them in-process, and snapshotting captured stdout. It also models shell-facing capabilities like the file system as dependency clients registered through `DependencyKey`/`DependencyValues`. That is the pattern this milestone adopts for `MicroSwiftCLI`. ([GitHub][1])

### 14.0.1 Core architectural decisions

M0 locks in the following choices:

- one root Swift package as the source of truth
- one executable CLI target rooted at `AsyncParsableCommand`
- toolchain formatting and linting through `swift format` and `swift-format lint --strict`
- dependency injection through `swift-dependencies`
- snapshot testing and inline snapshots for CLI and golden outputs
- MLX command-line execution through an Xcode build plus `mlx-run` wrapper
- pre-commit hook as the enforcement point for local verification

The lint/format choice is intentionally minimal. `swift-format` documents both `format` and `lint`, and its `lint` subcommand supports `--strict` so warnings become a failing exit status. ([GitHub][2])

### 14.0.3 Package and dependency policy

Package dependencies in M0 should be minimal and explicit:

- `mlx-swift`
- `swift-argument-parser`
- `swift-dependencies`
- `swift-custom-dump`
- `swift-snapshot-testing`

No SwiftLint plugin package. No auxiliary node/python/ruby tooling. No CI-only dependencies.

This follows the same shape as `pfw`вЂ™s package: one executable target, one test target, `ArgumentParser` for command structure, `Dependencies` for effect control, `CustomDump` for readable deterministic state rendering, and inline snapshot testing in tests. ([GitHub][1])

### 14.0.4 CLI surface in M0

The initial CLI should expose only a minimal verified surface:

- `micro-swift doctor`
- `micro-swift seed dump`
- `micro-swift mlx-smoke`

The root command should be `AsyncParsableCommand`. That matches both the intended future async nature of MLX orchestration and the concrete shape used by `pfw`, whose root command is declared as `@main struct PFW: AsyncParsableCommand`. ([GitHub][3])

### 14.0.5 Dependency clients in M0

M0 should define dependency clients for:

- `fileSystem`
- `env`
- `clock`
- `uuid`
- `process`
- `mlxRuntime`
- `stdout`
- `stderr`

Each client gets:

- a protocol or callable client type
- a `DependencyKey`
- a `DependencyValues` accessor
- a live implementation
- a test implementation

The design target is exactly the pattern visible in `pfw`вЂ™s dependency layer: file-system behavior is abstracted behind a protocol, registered as a `DependencyKey`, and surfaced through `DependencyValues.fileSystem`. ([GitHub][4])

### 14.0.6 Local verification commands

`Scripts/dev` is the only user-facing orchestration entrypoint. It should expose:

```sh
./dev doctor
./dev format
./dev lint
./dev build
./dev test
./dev mlx-smoke
./dev verify
./dev install-hooks
./dev run micro-swift <args...>
```

Command meanings:

- `doctor`: print toolchain/runtime diagnosis
- `format`: run formatter in write mode
- `lint`: run `swift-format lint --strict`
- `build`: `swift build`
- `test`: `swift test`
- `mlx-smoke`: build with `xcodebuild`, run through `Scripts/mlx-run`
- `verify`: the canonical local gate
- `install-hooks`: wire `.githooks/pre-commit`
- `run`: convenience entrypoint into the built CLI

The lint command should be implemented with the formatterвЂ™s lint subcommand rather than an external plugin lane, because `swift-format` already supports recursive and strict linting directly. ([GitHub][2])

### 14.0.7 Point-Free-style test harness

The CLI test harness should copy the core `pfw` pattern:

- parse commands with `MicroSwift.parseAsRoot(arguments)`
- execute them in-process
- capture stdout/stderr
- assert inline snapshots for user-facing output
- use a base Swift Testing suite that installs default dependencies
- use per-test or per-suite dependency overrides for special cases

`pfw` does exactly this in its command assertion helper: it parses the root command with `PFW.parseAsRoot(arguments)`, runs either the async or sync command path, captures stdout, and snapshot-tests the resulting lines. ([GitHub][5])

### 14.0.8 Snapshot strategy in M0

M0 snapshot coverage should already include:

- `doctor --json`
- `doctor` human-readable output
- `seed dump --json`
- `--help`
- `mlx-smoke --json`
- dependency-derived path rendering where relevant

Inline snapshots are preferred for short command outputs. File snapshots are preferred for structured or multiline artifacts.

The reason to formalize this immediately is that snapshots become the stable approval surface for every later compiler-facing text artifact: token tapes, IR dumps, diagnostics, corpus manifests, and eventually Wasm metadata dumps.

### 14.0.9 Deterministic benchmark seed infrastructure

M0 should already establish the seed contract for later corpora.

`Config/bench-seeds.json` should define:

- schema version
- global seed
- per-corpus seed ids

`MicroSwiftSpec` should define a typed manifest:

- `BenchSeedManifest`
- `CorpusID`
- deterministic derivation rules

All later benchmark generators must derive from this seed contract, not invent their own RNG setup.

### 14.0.10 MLX smoke path

The MLX smoke test proves the local runtime path works. It must:

1. build the CLI with `xcodebuild`
2. run the built CLI through `Scripts/mlx-run`
3. execute a trivial MLX kernel
4. emit stable JSON

This is not an arbitrary preference. The MLX Swift README says command-line SwiftPM cannot build the Metal shaders and that `xcodebuild` is the command-line build path for those binaries, while the MLX examples repo documents `mlx-run` as the command-line wrapper that locates and runs the Xcode-built executables equivalently to running from Xcode. ([GitHub][6])

### 14.0.11 Pre-commit hook contents

The pre-commit hook should be thin:

```sh
#!/usr/bin/env bash
set -euo pipefail
exec ./dev verify
```

No duplicated logic inside the hook. The hook delegates entirely to the canonical verification command.

### 14.0.12 Acceptance criteria

M0 is complete only when all of the following are true:

1. A fresh clone can run `./dev install-hooks` once and then rely on the checked-in pre-commit hook.
2. `./dev verify` exits `0` locally on a supported Apple silicon setup.
3. `micro-swift doctor --json` is deterministic except for explicitly marked volatile fields.
4. `micro-swift seed dump --json` matches a checked-in snapshot.
5. `micro-swift mlx-smoke --json` succeeds through the Xcode plus wrapper path.
6. The CLI tests run commands in-process via `parseAsRoot`, not through subprocess shells.
7. Snapshot tests cover both human-readable and structured output.
8. The full local verification path stays within the latency budget.

### 14.0.13 Deliverables

- root Swift package with all seven modules
- `AsyncParsableCommand` CLI scaffold
- dependency client layer for shell effects
- `Scripts/dev`
- `Scripts/mlx-run`
- `.githooks/pre-commit`
- `./dev install-hooks`
- initial CLI commands: `doctor`, `seed dump`, `mlx-smoke`
- inline snapshot harness and base suite
- deterministic seed manifest
- ADRs and milestone documentation

### 14.0.14 Why this milestone matters

This milestone encodes the shell architecture before the compiler exists. If M0 is wrong, every later milestone inherits friction: untestable command surfaces, ad hoc environment reads, nondeterministic output, and flaky MLX execution.

The purpose of M0 is to make those invalid states structurally harder to represent:

- all shell effects are dependencies, not ambient globals
- all user-facing outputs are snapshotted
- all local verification funnels through one canonical command
- MLX runtime setup is captured in a checked-in wrapper rather than tribal knowledge
- pre-commit enforces the same verification path every contributor uses manually

Under the constitution, Milestone 0 is not вЂњthe repo exists.вЂќ It is вЂњa fresh clone has one canonical local proof path, one canonical MLX runtime path, deterministic output surfaces, and a shell architecture that keeps later milestones honest.вЂќ
