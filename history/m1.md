Below is a copy-pasteable expansion of **Milestone 1** that is tighter than the draft in two important ways:

1. The page summary should carry **`lineBreakCount`**, not only `lineCount`. `lineCount` is ambiguous at EOF and around trailing newlines; `lineBreakCount` is additive and prefix-scan friendly.
2. The canonical line resolver should be defined over **byte offsets, including EOF**, because spans are half-open and diagnostics will eventually need to point at `end == byteCount`.

## Milestone 1 â€” Source tape, spans, and file paging

### Goal

Milestone 1 establishes the compilerâ€™s canonical source representation. It turns an input file into an immutable, deterministic, byte-precise value that every later phase can consume without reinterpreting text or reconstructing provenance.

The output of this milestone is not â€śsome bytes plus helper functions.â€ť It is a **prepared source artifact** with:

- exact raw bytes, preserved byte-for-byte
- compact line-start metadata for line/column lookup
- validated half-open spans
- a deterministic newline-aligned page plan
- additive page summaries suitable for later prefix scans

This is the first place where the compilerâ€™s correctness discipline becomes real. If Milestone 1 is loose, every later diagnostic, token span, and benchmark is contaminated.

### Constitutional interpretation

This milestone should explicitly follow the constitution:

**Correctness by construction.**
Source locations use fixed-width offset types and half-open spans. Invalid spans are impossible to construct except through a validating initializer.

**Functional core, imperative shell.**
Filesystem IO lives in a thin loader. Everything after `Data` acquisition is a pure transformation:
`bytes -> line structure -> page plan -> prepared source`.

**Verifiable rewards.**
Every subcomponent has an executable oracle:

- scalar reference implementation in tests
- production implementation
- parity tests between them
- golden snapshots for dumps

**Density and minimal dependencies.**
Do not pull in a property-testing dependency for this. A 30â€“50 line deterministic PRNG in the test target is enough.

**Living documentation.**
The source-format invariants are documented in repo and asserted in tests. Any change to newline semantics or page planning breaks goldens.

## Scope

Milestone 1 includes:

- loading a source file as exact bytes
- deriving line terminator structure
- deriving compact line-start offsets
- validating and resolving spans
- deterministic paging for giant files
- stable JSON / textual dumps for snapshots

Milestone 1 does **not** include:

- lexing
- Unicode-aware display columns
- string decoding or normalization
- incremental edits
- custom Metal kernels
- MLX compile-time optimization work

Milestone 1 should stay in pure Swift plus MLX array operations. Current MLX Swift documentation still notes that command-line SwiftPM cannot build Metal shaders and that `xcodebuild` can, so this milestone should not introduce any custom Metal dependency. MLXâ€™s compile documentation also emphasizes first-call compilation overhead and recompilation when input shape or type changes, which is another reason not to optimize Milestone 1 around compiled kernels yet; fixed-shape page-local compilation belongs later. Swift Testing is suitable for the acceptance suite because Apple positions it for testing Swift packages and Xcode projects. ([GitHub][1])

## Canonical data model

The key design choice here is:

> **Persist the compact representation canonically.**
> Dense per-byte line-id arrays are derivable, not foundational.

That matters for giant files. A mandatory dense `Int32` line-id vector is expensive. The compact line-start table is the true source of truth.

### Fixed-width identity types

```swift
public struct FileID: RawRepresentable, Hashable, Sendable {
    public let rawValue: UInt32
    public init(rawValue: UInt32) { self.rawValue = rawValue }
}

public struct ByteOffset: RawRepresentable, Comparable, Hashable, Sendable {
    public let rawValue: Int64
    public init(rawValue: Int64) { self.rawValue = rawValue }
    public static func < (lhs: Self, rhs: Self) -> Bool { lhs.rawValue < rhs.rawValue }
}

public struct LineIndex: RawRepresentable, Comparable, Hashable, Sendable {
    public let rawValue: Int32
    public init(rawValue: Int32) { self.rawValue = rawValue }
    public static func < (lhs: Self, rhs: Self) -> Bool { lhs.rawValue < rhs.rawValue }
}

public struct ColumnIndex: RawRepresentable, Comparable, Hashable, Sendable {
    public let rawValue: Int32
    public init(rawValue: Int32) { self.rawValue = rawValue }
    public static func < (lhs: Self, rhs: Self) -> Bool { lhs.rawValue < rhs.rawValue }
}
```

No `Int`, no `UInt`, no platform-sized offsets in serialized structures.

### Spans

All spans are half-open: `[start, end)`.

```swift
public struct Span: Hashable, Sendable {
    public let fileID: FileID
    public let start: ByteOffset
    public let end: ByteOffset
}
```

Rules:

- `0 <= start <= end <= byteCount`
- zero-width spans are allowed
- spans never cache line/column
- line/column is always derived from the prepared source

A span is only constructed by a validating API. Do not expose a raw public initializer that permits invalid bounds.

### Source buffer vs source tape

Use a two-view model.

```swift
public struct SourceBuffer: Sendable {
    public let fileID: FileID
    public let path: String
    public let bytes: Data   // exact raw bytes, never normalized
}

public struct SourceTape: Sendable {
    public let fileID: FileID
    public let byteCount: Int64

    public let bytes: MLXArray
    // dtype: uint8, shape: [N]

    public let lineStartOffsets: MLXArray
    // dtype: int64, shape: [L]

    public let lineCount: Int32
}
```

`SourceBuffer` is the canonical host-side view for round-tripping and future diagnostic rendering.
`SourceTape` is the canonical tensor-side view for later data-parallel phases.

### Compact host index

Keep a compact host mirror for cheap line/column lookup and page planning:

```swift
public struct HostLineIndex: Sendable {
    public let lineStartOffsets: [Int64]   // strictly increasing, always starts with 0
    public let lineCount: Int32
}
```

This is intentionally compact and deterministic.

### Prepared source

```swift
public struct PreparedSource: Sendable {
    public let buffer: SourceBuffer
    public let tape: SourceTape
    public let hostLineIndex: HostLineIndex
    public let pages: [SourcePage]
}
```

This is the sole output of Milestone 1.

## Newline semantics

This needs to be specified now, not discovered later.

### What counts as a line break

A line break is one of:

- `LF` (`0x0A`)
- lone `CR` (`0x0D` not followed by `LF`)
- `CRLF`, counted as a **single** line break whose terminal byte is the `LF`

This gives deterministic behavior across files from different environments while preserving exact bytes.

### What is a line

A logical line begins at offset `0` or immediately after a recognized line break.

`lineStartOffsets` therefore satisfies:

- it always contains `0`
- it is strictly increasing
- each entry is in `0...byteCount`
- it contains `byteCount` iff the file ends with a line terminator
- `lineCount == lineStartOffsets.count`

Examples:

- empty file `""`
  `lineStartOffsets = [0]`, `lineCount = 1`

- `"a"`
  `lineStartOffsets = [0]`, `lineCount = 1`

- `"a\n"`
  `lineStartOffsets = [0, 2]`, `lineCount = 2`

- `"a\r\nb"`
  `lineStartOffsets = [0, 3]`, `lineCount = 2`

### Columns

Columns are **byte columns**, zero-based internally.

That means:

- no grapheme accounting
- no tab expansion
- no display-width logic

User-facing diagnostics can render one-based columns later, but the stored truth is byte-based.

## Tensor algorithms

Milestone 1 should have a scalar oracle and a production implementation. The scalar oracle lives only in tests. The production path may use MLX array operations, but the observable result must match the oracle exactly.

### 1. Line terminator end mask

Given `bytes[0..<N]`, compute `lineTerminatorEndMask[0..<N]` where `true` means â€śthis byte is the last byte of a line terminator.â€ť

Pseudocode:

```swift
isLF[i] = bytes[i] == 0x0A
isCR[i] = bytes[i] == 0x0D
nextIsLF[i] = (i + 1 < N) && (bytes[i + 1] == 0x0A)

lineTerminatorEndMask[i] =
    isLF[i] || (isCR[i] && !nextIsLF[i])
```

This makes `CRLF` contribute exactly one line break at the `LF`.

### 2. Line-start offsets

Canonical compact derivation:

```swift
lineStartOffsets = [0] + [i + 1 for i in 0..<N where lineTerminatorEndMask[i]]
```

This is the canonical representation of line structure.

### 3. Optional dense line-index-by-offset

Do **not** store this by default. Provide it as a derived helper for tests and future kernels that truly need it.

Conceptually:

```swift
lineIndexByOffset[o] = upperBound(lineStartOffsets, o) - 1
// for o in 0...N
```

This includes EOF and makes half-open span resolution precise.

### 4. Location resolution

```swift
func resolve(_ offset: ByteOffset, in source: PreparedSource) -> SourceLocation
```

Definition:

- `line0 = upperBound(lineStartOffsets, offset) - 1`
- `column0 = offset - lineStartOffsets[line0]`

This must work for `offset == byteCount`.

### 5. Span validation

```swift
func makeSpan(
    fileID: FileID,
    start: ByteOffset,
    end: ByteOffset,
    in source: PreparedSource
) throws -> Span
```

Reject:

- negative offsets
- `start > end`
- `end > byteCount`
- mismatched `fileID`

## Deterministic page planning

This is the other core deliverable.

### Page policy

```swift
public struct PagePolicy: Hashable, Sendable {
    public let targetBytes: Int32
}
```

For Milestone 1, keep policy intentionally narrow. Later milestones can add tuning fields.

Validation:

- `targetBytes > 0`

### Fundamental rule

**Pages are newline-aligned or EOF-aligned.**

Milestone 1 should not split in the middle of a logical line. That keeps page state trivial for the later lexer design, which currently assumes newline-aligned pages.

### Page type

```swift
public struct SourcePage: Hashable, Sendable {
    public let pageID: Int32
    public let start: ByteOffset
    public let end: ByteOffset

    public let byteCount: Int32
    public let lineBreakCount: Int32
    public let oversize: Bool
}
```

`lineBreakCount` is additive. That is the field later prefix scans actually want.

### Page planning algorithm

For non-empty sources:

1. `start = 0`
2. `target = min(start + targetBytes, byteCount)`
3. Choose `end` as:
   - the greatest line start `<= target` and `> start`, if one exists
   - otherwise the smallest line start `> target`
   - otherwise `byteCount`

4. Emit page `[start, end)`
5. Set `start = end`
6. Repeat until `start == byteCount`

For the empty file, emit exactly one empty page:

- `pageID = 0`
- `start = 0`
- `end = 0`
- `byteCount = 0`
- `lineBreakCount = 0`
- `oversize = false`

This avoids a special â€śno pagesâ€ť state.

### Oversize pages

A page is `oversize` when `byteCount > targetBytes`.

This is not an error. It is expected when a single logical line exceeds the target page size.

### Additive summary invariant

For pages `p0 ... pk-1`:

- pages are contiguous
- pages are non-overlapping
- union of page spans is exactly `[0, byteCount)`
- `sum(page.byteCount) == byteCount`
- `sum(page.lineBreakCount) == globalLineCount - 1`

That last invariant is why `lineBreakCount` is the right additive field.

## Why this representation is correct

There are a few invariants worth stating explicitly because they will later support lexing, diagnostics, and benchmarking.

### Invariant A â€” bytes are never normalized

No BOM stripping.
No newline rewriting.
No UTF-8 decoding.
No text transcoding.

Milestone 1 is byte-exact.

### Invariant B â€” spans are stable under later phases

Everything later refers back to source by `(fileID, start, end)`.
Nothing in later phases is permitted to reinterpret source length, line numbering, or page boundaries.

### Invariant C â€” paging is a pure function

`pages = pagePlan(bytes, lineStarts, policy)`

No filesystem metadata, locale, wall clock, hash-table iteration, or host-specific behavior may affect the plan.

### Invariant D â€” line lookup has one truth

Host lookup and tensor lookup must agree exactly.
There is no â€śprettyâ€ť line model and â€ścompilerâ€ť line model.

## Testing strategy

Milestone 1 should be unusually well tested because every later phase depends on it.

### 1. Scalar oracle

In the test target, implement a tiny scalar reference over `[UInt8]`:

- line terminator detection
- line-start collection
- offset resolution
- page planning

This oracle is allowed to be boring. It should prioritize obvious correctness over speed.

### 2. Production parity tests

For every fixture and random corpus:

- scalar oracle output
- production output
- exact equality

### 3. Edge-case fixtures

Check in explicit fixtures for:

- empty file
- one-byte file
- no trailing newline
- trailing newline
- multiple blank lines
- LF only
- CRLF only
- lone CR
- huge single line
- long final line without newline
- bytes containing `0x00`
- non-ASCII UTF-8 sequences preserved as raw bytes

### 4. Property tests

Use a tiny deterministic PRNG in repo. Generate random byte arrays and page sizes.

Properties:

- `lineStartOffsets.first == 0`
- `lineStartOffsets` strictly increasing
- all offsets are in `0...byteCount`
- `lineCount == 1 + numberOfLineTerminators`
- `resolve(offset)` agrees with oracle for all sampled offsets, including EOF
- pages cover the file with no gaps or overlaps
- every page end is either EOF or a line start
- `sum(page.byteCount) == byteCount`
- `sum(page.lineBreakCount) == lineCount - 1`

### 5. Determinism soak tests

Run the same input and page policy repeatedly and compare serialized snapshot digests.

At minimum:

- one small fixture repeated many times
- one large fixture repeated several times

### 6. Golden snapshot tests

Golden-test:

- canonical JSON snapshot
- textual debug snapshot

The tests should fail on any semantic drift in line or page behavior.

### 7. No hidden mutable state

A repeated `load -> dump` cycle on the same input must produce byte-for-byte identical dumps.

## Automated reward loops

This milestone is a perfect place to enforce the â€śverifiable rewardsâ€ť rule.

Every subtask should end with a binary check:

1. **Round-trip reward**
   Raw bytes loaded from disk equal original fixture bytes.

2. **Line-table reward**
   Production line starts equal oracle line starts.

3. **Span reward**
   Valid spans construct; invalid spans reject.

4. **Paging reward**
   Page plan passes coverage and additive invariants.

5. **Determinism reward**
   Snapshot digest is identical across repeated runs.

Then the review order should be:

1. machine checks
2. AI judge over API and invariant clarity
3. human architectural review

Human review should not be spending time checking whether line 37 resolves to column 12. That should already be proven by tests.

## Acceptance criteria

Milestone 1 is done when all of the following hold.

### Functional correctness

- source loads byte-for-byte with no normalization
- line-start offsets are correct for LF, CRLF, lone CR, trailing newline, and empty-file cases
- arbitrary byte offsets, including EOF, resolve to correct line/column pairs
- validated spans reject invalid bounds
- page plans are newline-aligned and deterministic
- page summaries satisfy additive invariants

### Determinism

- identical input bytes and page policy always produce identical:
  - `lineStartOffsets`
  - page boundaries
  - page summaries
  - debug snapshots

### Architecture

- shell concerns are isolated to file IO and CLI
- pure transforms are testable without touching the filesystem
- page planning depends only on `lineStartOffsets`, `byteCount`, and `PagePolicy`
- no hidden reliance on locale, decoded strings, or nondeterministic hash iteration

### Quality gates

- all Milestone 1 tests pass
- snapshots are checked into repo
- docs describing newline and paging semantics are checked into repo
- the total test suite remains comfortably within the projectâ€™s latency budget
