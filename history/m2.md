I incorporated the Swift-embedded DSL change plus the runtime-bridge refinements that matter for Milestone 3: explicit class sets, stable token-kind IDs, flattened fallback transition tables, sorted class-set membership, and bounded-lookahead metadata.

# Milestone 2 — Embedded Swift Regex DSL and Lexer Generator

## Objective

Turn the Micro-Swift lexer from an informal set of token rules into a **declarative, typed, versioned Swift specification** that compiles into a **stable lexer artifact** consumed by later runtime milestones.

This milestone ends at **artifact generation**. It does **not** implement tensor lexing kernels yet. The output is a pure data product:

```text
Swift LexerSpec value
  -> DeclaredSpec
  -> NormalizedSpec
  -> ValidatedSpec
  -> ByteClasses
  -> ClassSets
  -> ClassifiedRules
  -> LexerArtifact
```

Each arrow is a pure function with golden tests and property tests. The only imperative shell is the CLI or build-plugin entrypoint that loads a checked-in Swift spec value and writes the generated artifact.

That separation is important. It gives you:

- correctness by construction through typed intermediate forms
- deterministic outputs suitable for golden approval
- a narrow contract for Milestone 3
- fast verification loops with no GPU runtime involved yet

The authoring DSL is embedded in Swift, but the **review surface remains the checked-in canonical JSON artifact**. The Swift DSL is how the spec is written; the artifact is how it is reviewed, diffed, and consumed.

---

## Scope

### In scope

Milestone 2 owns:

- the embedded Swift lexer DSL
- declaration lowering from Swift DSL values into typed IR
- AST normalization
- regex validation
- byte equivalence class construction
- canonical class-set construction over byte classes
- rule-family classification
- deterministic fallback automaton construction for residual rules
- canonical artifact serialization
- generator diagnostics
- build integration and golden tests

### Explicitly out of scope

Milestone 2 does **not** own:

- MLX runtime execution
- tokenization of source programs
- longest-match runtime resolution
- span production for source files
- custom Metal kernels
- source-language diagnostics

Those begin in Milestones 3 and 4.

## Design constraints

This milestone should follow six hard rules.

### 1. The generator is pure

The core generator is a pure library:

```swift
compile(_ spec: LexerSpec, options: CompileOptions) throws -> LexerArtifact
```

No global state. No timestamps. No randomization. No host-path leakage into outputs.

### 2. The authoring DSL is embedded in Swift, but the spec itself is still a pure value

The lexer spec is not a free-form program. It is a closed Swift value built from deterministic constructors.

That means:

- no filesystem reads
- no environment-variable reads
- no time or randomness
- no data-dependent branching on host state
- no iteration over unordered collections
- no generation from external files at compile time

The checked-in language spec should be a top-level `let` composed from constants and builder calls only.

### 3. Invalid states are unrepresentable where Swift can make them so

Do not carry loosely-typed dictionaries or ad hoc JSON blobs internally.

Use distinct types for:

- declared regex values
- normalized regex values
- validated token rules
- class sets
- classified rule families
- fallback DFAs
- final artifacts

Use the Swift type system aggressively in the authoring DSL:

- token kinds are typed symbols or enum cases, not loose strings
- keyword remaps attach only to an `IdentifierRuleHandle`
- unsupported regex features are not representable because the DSL does not expose constructors for them
- malformed ranges and custom escape bugs disappear because the DSL uses typed byte sets and ordinary Swift literals instead of a separate text parser

Semantic invariants that cannot be carried comfortably at the type level still belong to validation. A rule that can match the empty string must be rejected before it can become a runtime artifact. A rule whose lowered machine exceeds configured caps must never inhabit the final artifact.

### 4. One-way data flow

The pipeline is strictly:

```text
Swift LexerSpec value
-> declared spec
-> normalized canonical regex IR
-> validated rule set
-> byte classes
-> class sets
-> rule-family lowering
-> canonical artifact
```

No stage mutates prior-stage data in place. Every stage emits a new value.

### 5. Formal checks before review

Every stage gets automated checks:

- declaration-lowering golden tests
- normalization idempotence tests
- byte-class partition tests
- class-set canonicalization tests
- compile-twice-equals-byte-for-byte tests
- reference-matcher equivalence tests
- artifact round-trip tests

Human review is for intent and shape. “Did the generator do the right thing?” should already be answered mechanically.

### 6. Reproducibility is part of the artifact contract

Two runs over the same checked-in spec on two machines must produce identical artifact bytes.

That implies:

- no timestamps
- no nondeterministic map iteration
- no host-specific paths
- stable sorting everywhere
- canonical serialization
- `specHashHex` computed from the canonical semantic spec, not raw Swift source text

## User-facing DSL

The DSL is intentionally small. It is not a general regex language. It is a Swift-embedded lexer-spec language for a byte-oriented compiler.

### Example

```swift
import MicroSwiftLexerGen

enum MicroSwiftToken: String, CaseIterable, Sendable {
  case ws
  case lineComment

  case ident
  case int

  case arrow
  case eqEq
  case eq
  case plus
  case minus
  case star
  case slash
  case lParen
  case rParen
  case lBrace
  case rBrace
  case colon
  case comma

  case kwFunc
  case kwLet
  case kwReturn
  case kwIf
  case kwElse
  case kwTrue
  case kwFalse
}

let MicroSwiftV0 = LexerSpec(name: "MicroSwift.v0") {
  skip(.ws, oneOrMore(.asciiWhitespace))
  skip(.lineComment, .literal("//") <> zeroOrMore(.not(.newline)))

  let ident = identifier(
    .ident,
    .asciiIdentStart <> zeroOrMore(.asciiIdentContinue)
  )

  token(.int, oneOrMore(.asciiDigit))

  token(.arrow, .literal("->"))
  token(.eqEq,  .literal("=="))
  token(.eq,    .literal("="))
  token(.plus,  .literal("+"))
  token(.minus, .literal("-"))
  token(.star,  .literal("*"))
  token(.slash, .literal("/"))
  token(.lParen, .literal("("))
  token(.rParen, .literal(")"))
  token(.lBrace, .literal("{"))
  token(.rBrace, .literal("}"))
  token(.colon, .literal(":"))
  token(.comma, .literal(","))

  keywords(for: ident) {
    keyword("func",   as: .kwFunc)
    keyword("let",    as: .kwLet)
    keyword("return", as: .kwReturn)
    keyword("if",     as: .kwIf)
    keyword("else",   as: .kwElse)
    keyword("true",   as: .kwTrue)
    keyword("false",  as: .kwFalse)
  }
}
```

The exact surface syntax may vary, but these properties are required:

- the spec is ordinary Swift
- it is built from a narrow, deterministic API
- token kinds are typed
- keyword remaps accept only an identifier-like rule handle
- regex construction is byte-oriented

### Supported constructs

The DSL should support only these regular constructs, expressed as Swift value constructors and combinators:

- ASCII byte-string literals, for example `.literal("func")`, `.literal("->")`
- raw byte literals, if needed, for example `.bytes([0x2D, 0x3E])`
- byte classes and byte sets, for example `.asciiDigit`, `.asciiIdentStart`, `.byteSet(...)`, `.not(.newline)`
- concatenation
- alternation
- repetition: zero-or-more, one-or-more, optional
- bounded repetition: `{m}`, `{m,n}`, `{m,}` semantics via an explicit repetition constructor
- grouping via ordinary Swift expression nesting

There is no separate custom escape syntax. Swift string literals handle their own escapes; explicit byte arrays handle arbitrary bytes.

### Unsupported constructs

These are not merely “rejected later”; they should be absent from the authoring surface entirely:

- backreferences
- lookaround
- named groups
- captures with semantic meaning
- Unicode categories
- anchors `^` and `$`
- dot `.` as an implicit wildcard

This keeps the language regular in the straightforward sense and avoids hidden complexity.

### Byte model

The DSL is defined over **raw bytes**, not Unicode scalars. For v0, identifiers are ASCII-only by construction. Non-ASCII source bytes can become lexical errors later; they do not need magical support in the regex language.

ASCII string literals in the DSL must either:

- be rejected if they contain non-ASCII scalars, or
- require the author to spell them as explicit bytes

That keeps semantics exact and byte-level.

## Static semantics of the DSL

Declaration construction is not enough. The generator needs a validation phase that enforces lexer invariants.

### Invariants the Swift type system should already enforce

The authoring DSL should make these impossible or awkward enough that they are not the default failure mode:

- keyword remaps targeting a non-identifier base rule
- unsupported regex features
- malformed DSL-level ranges
- malformed custom escape sequences, because there are none
- duplicate token kind spellings, if kinds are represented by enum cases or typed symbols rather than strings

### Required validation checks

A spec is invalid if any of these hold:

- a `token` or `skip` rule can match the empty string
- the same token kind is introduced by multiple top-level rules unless the DSL explicitly permits aliasing
- a keyword kind name is duplicated inside the same keyword table
- a keyword lexeme is duplicated in the same keyword table
- a keyword lexeme is not accepted by the base identifier rule
- a bounded repetition has `max < min`
- a rule lowers only to `fallback` while fallback is disabled
- a rule lowers to `localWindow` with width above `maxLocalWindowBytes`
- a fallback machine exceeds configured limits
- a finite rule width exceeds the artifact field width chosen for v0

### Recommended additional checks

These are valuable as hard errors or warnings:

- exact duplicate normalized regexes for distinct rules
- unreachable keyword remaps
- rules that are semantically identical after normalization
- rules whose fallback automata are close to configured caps

The last one is important. If a regex explodes into an unreasonable fallback machine, the correct outcome is not “hope the runtime copes.” The correct outcome is a build failure with a precise spec diagnostic.

### Bridge to Milestone 3

The generator may support all four rule families, but the **checked-in `MicroSwift.v0` spec must compile to a runtime profile already supported by Milestone 3**.

For the shipped v0 language profile, that means:

- top-level rules lower only to `literal` or `run`
- keywords lower only to remap tables
- the checked-in artifact must not require `localWindow` or `fallback` execution in the first runtime milestone

That keeps the artifact contract honest.

## Internal representation

Use typed IR rather than carrying declaration syntax forward.

### Declared IR

```swift
public struct DeclaredSpec: Sendable, Equatable {
  public let name: String
  public let rules: [DeclaredRule]
  public let keywordBlocks: [DeclaredKeywordBlock]
}

public struct DeclaredRule: Sendable, Equatable {
  public let declaredKindName: String
  public let mode: RuleMode
  public let role: RuleRole          // token, skip, identifierLike
  public let regex: RawRegex
  public let sourceSpan: SpecSpan
}
```

The exact public shape can vary, but the important point is that this stage is already a typed semantic declaration, not parsed text.

### Normalized IR

```swift
public struct NormalizedSpec: Sendable, Equatable {
  public let name: String
  public let rules: [NormalizedRule]
  public let keywordBlocks: [KeywordBlock]
}

public struct NormalizedRule: Sendable, Equatable {
  public let ruleID: RuleID
  public let tokenKindID: TokenKindID
  public let name: String
  public let mode: RuleMode
  public let role: RuleRole
  public let regex: NormalizedRegex
  public let props: RegexProps
}
```

### Regex properties

Every normalized regex should carry precomputed facts:

```swift
public struct RegexProps: Sendable, Equatable {
  public let nullable: Bool
  public let minWidth: Int
  public let maxWidth: Int?      // nil == unbounded
  public let firstByteSet: ByteSet
}
```

This is the first “correctness by construction” win. Most later decisions become total functions over `RegexProps`, not ad hoc pattern matching.

## Normalization

Normalization is the heart of this milestone. The goal is a **canonical regex IR** where semantically equivalent specs normalize to the same representation.

### Required normalization steps

1. Desugar API-level sugar.
   - one-or-more becomes `{1,}`
   - optional becomes `{0,1}`
   - byte-set helpers become explicit byte sets
   - ASCII helpers become explicit byte sets

2. Flatten associative structure.
   - nested concatenations become a single concat node
   - nested alternations become a single union node

3. Eliminate neutral elements.
   - concat with epsilon removes epsilon
   - union with duplicates removes duplicates

4. Merge adjacent literals.
   - adjacent literal byte strings become one literal node

5. Canonicalize alternation ordering.
   - children sorted by canonical structural key, not source expression order

6. Canonicalize byte classes.
   - byte classes stored as sorted non-overlapping ranges or a 256-bit mask
   - negated classes reduced to explicit byte sets over `0...255`

7. Compute properties.
   - nullable
   - min/max width
   - first-byte set

### Idempotence requirement

Normalization must be idempotent:

```text
normalize(normalize(x)) == normalize(x)
```

This should be a property test, not an informal hope.

### Why this matters

Without canonicalization, deterministic state numbering is fragile. With canonicalization, fallback automata can use structural identity safely, duplicate rules become detectable, and golden diffs become meaningful.

## Byte equivalence classes

This is the first bridge between declarative regexes and fast runtime lexing.

### Goal

Collapse the 256-byte alphabet into a smaller set of equivalence classes such that bytes in the same class are indistinguishable with respect to every normalized rule.

### Construction algorithm

Compute the set of primitive byte predicates appearing in the normalized spec:

- every singleton byte in literals
- every explicit byte set used anywhere
- every `firstByteSet`
- every stop set and body set that will matter during lowering

Then compute, for each byte `b ∈ 0...255`, a signature:

```text
signature(b) = [b ∈ P0, b ∈ P1, ..., b ∈ Pn]
```

Bytes with identical signatures are equivalent and belong to the same byte class.

### Deterministic numbering

Number byte classes by:

1. smallest member byte ascending
2. then full member list lexicographically

This is stable and total.

### Output

The artifact should include both:

- `byteToClass[256]`
- `classes[classID].bytes`

The first is for runtime classification. The second is for debugging, artifact diffs, and diagnostics.

### Invariant

For every normalized rule and every two bytes `x`, `y` in the same class:

```text
substitute(x for y) does not change any rule transition behavior
```

You do not need to prove that formally, but the construction makes it true by definition.

## Canonical class sets

Milestone 3 needs more than byte classes. It needs reusable **sets of byte classes** that correspond to semantic predicates like digit, identifier-start, identifier-continue, not-newline, first-byte sets, and rule-local stop sets.

This stage makes those explicit.

### Goal

For every byte set that still matters after normalization and lowering, compute its quotient over byte classes and assign it a stable `ClassSetID`.

### Construction algorithm

Collect every relevant byte set:

- rule `firstByteSet`
- every body/head/tail/stop set that will appear in `RunPlan`
- every local-window transition predicate
- every fallback transition predicate if needed

Project each byte set through `byteToClass` to get a set of class IDs:

```text
classSet(S) = { c | some byte in class c belongs to S }
```

Deduplicate identical class-ID sets.

### Deterministic numbering

Number class sets by lexicographic ordering of their sorted member class IDs.

### Output

The artifact should include:

- `classSets[classSetID].classes`

Every runtime plan must refer only to:

- raw literal bytes
- byte class IDs
- class set IDs

No runtime stage should have to reconstruct semantic byte predicates from regex source.

### Invariant

For any original byte set `S` used in lowering and any byte `b`:

```text
b ∈ S    iff    byteToClass[b] ∈ classSet(S)
```

That is the invariant Milestone 3 will rely on when it turns per-byte `classID` values into masks.

## Rule-family classification

The generator must lower each validated rule into exactly one runtime family.

```swift
public enum RuleFamily: String, Codable, Sendable {
  case literal
  case run
  case localWindow
  case fallback
}
```

This type is a good example of making invalid states harder to represent. Later runtime code pattern-matches over a closed set of families rather than guessing.

### 1. Literal

A rule is `literal` iff its normalized form is exactly one fixed byte string.

Examples:

- `"=="`
- `"->"`
- `"("`

Lowered plan:

```swift
public struct LiteralPlan: Sendable, Codable, Equatable {
  public let bytes: [UInt8]
}
```

### 2. Run

A rule is `run` iff it is one of the supported repetitive shapes that can be evaluated with boundary masks and run-length logic.

At minimum support these subshapes:

- `C+`
- `H T*`, with invariant `H ⊆ T`
- `P B*` where `P` is a fixed literal prefix and `B` is a body class set

That covers:

- whitespace
- integers
- identifiers
- line comments

A concrete internal shape is:

```swift
public enum RunPlan: Sendable, Codable, Equatable {
  case classRun(bodyClassSetID: UInt16, minLength: UInt16)
  case headTail(headClassSetID: UInt16, tailClassSetID: UInt16)
  case prefixed(prefix: [UInt8], bodyClassSetID: UInt16, stopClassSetID: UInt16?)
}
```

For line comments, `prefix = ["/", "/"]`, `body = notNewline`, `stop = newline`.

### 3. Local window

A rule is `localWindow` iff it has finite maximum width and that width is at most `maxLocalWindowBytes`.

This family is for bounded regexes that are not pure literals but still small enough to check with local shifted masks or a bounded automaton.

The artifact should encode a tiny bounded automaton, not a giant list of all strings.

```swift
public struct LocalWindowPlan: Sendable, Codable, Equatable {
  public let maxWidth: UInt8
  public let stateCount: UInt16
  public let startState: UInt16
  public let acceptStates: [UInt16]
  public let transitions: [WindowTransition]   // over byte classes
}
```

### 4. Fallback

Anything not handled by the first three families becomes `fallback`, provided fallback is enabled.

Fallback must be deterministic. The simplest principled route is:

- derivative-based DFA over **byte classes**, not raw bytes
- canonical state keys from normalized regex
- discovery by BFS over ascending class IDs

```swift
public struct FallbackPlan: Sendable, Codable, Equatable {
  public let stateCount: UInt32
  public let classCount: UInt16
  public let transitionRowStride: UInt16
  public let startState: UInt32
  public let acceptingStates: [UInt32]
  public let transitions: [UInt32]   // dense row-major: stateCount * transitionRowStride
}
```

The canonical JSON artifact must store fallback transitions as a **dense flattened matrix**, not a jagged array. Transition lookup is:

```text
nextState = transitions[state * transitionRowStride + classID]
```

For the canonical artifact, `transitionRowStride` should equal `classCount`. Any extra padding for MLX- or Metal-friendly alignment belongs in the runtime bridge, not in the normative JSON artifact.

### Classification order

Classification must be greedy and deterministic:

1. literal
2. run
3. localWindow
4. fallback

The generator should never choose `fallback` for something representable as a cheaper family.

It should also prefer `run` over `localWindow` whenever a rule satisfies both classifications. Bounded repetition does not justify abandoning the cheaper run-family lowering.

## Fallback automaton construction

This is the only part of Milestone 2 with meaningful automata theory, so it deserves to be explicit.

### State representation

A DFA state is a canonical normalized regex derivative with respect to consumed input.

Use a stable key:

```text
canonicalKey(stateRegex) = canonical serialization of normalized regex
```

### Numbering policy

Assign state IDs as follows:

- state 0: dead state
- state 1: start state
- remaining states: BFS discovery order
- successor enumeration order: byte class ID ascending

That gives fully deterministic state numbering.

### Acceptance

A state is accepting iff its derivative regex is nullable.

### Resource limits

Fallback construction must have explicit limits:

- `maxFallbackStatesPerRule`
- `maxFallbackTransitions`
- optional derivative-depth safety valves only if actually needed

Exceeding a limit is a generator failure with a precise spec diagnostic.

Do not silently degrade.

## Keywords are remaps, not hot-path regexes

This should be a firm milestone rule.

Keywords must not be compiled as ordinary top-level regex rules. They are emitted as deterministic remap tables attached to a designated identifier-like base rule.

### Why

It avoids unnecessary overlap in the hot path:

- `func` is first lexed as `IDENT`
- then remapped to `KW_FUNC`

That gives cleaner longest-match semantics and reduces rule interference.

### Keyword artifact

Store keyword tables keyed by their base rule.

Each table should contain:

- `baseRuleID`
- `baseTokenKindID`
- `maxKeywordLength`
- entries sorted by:
  1. byte length descending
  2. then lexeme bytes ascending

That is total and deterministic.

The runtime can use either binary search on a lexicographically organized table or a compact trie built from the same sorted input. For Milestone 2, the artifact only needs one canonical representation.

## Artifact format

The artifact should be **normative JSON**, with an optional generated Swift wrapper later. JSON is the better authority because it is diffable and language-neutral.

### Canonical serialization rules

Use a custom canonical writer. Do not rely on default encoder map ordering.

Rules:

- fixed field order
- arrays already in semantic order
- no pretty-printer randomness
- integers only, no floats
- UTF-8 JSON
- no timestamps
- no absolute paths
- map-like data encoded as sorted arrays, not JSON objects with arbitrary key order
- include format version and semantic spec hash

`specHashHex` should be the hash of the canonical semantic spec representation after normalization and validation, not a hash of raw Swift source text. Equivalent specs with different surface spelling should converge.

### Required top-level fields

```swift
public struct LexerArtifact: Sendable, Codable, Equatable {
  public let formatVersion: UInt32
  public let specName: String
  public let specHashHex: String
  public let generatorVersion: String

  public let runtimeHints: RuntimeHints
  public let tokenKinds: [TokenKindDecl]
  public let byteToClass: [UInt8]          // length 256
  public let classes: [ByteClassDecl]
  public let classSets: [ClassSetDecl]
  public let rules: [LoweredRule]
  public let keywordRemaps: [KeywordRemapTable]
}
```

### Runtime hints

```swift
public struct RuntimeHints: Sendable, Codable, Equatable {
  public let maxLiteralLength: UInt16
  public let maxBoundedRuleWidth: UInt16
  public let maxDeterministicLookaheadBytes: UInt16
}
```

Definitions:

- `maxLiteralLength` is the maximum byte length over all `literal` rules
- `maxBoundedRuleWidth` is the maximum finite `maxWidth` over all non-fallback rules with finite width
- `maxDeterministicLookaheadBytes` is the maximum fixed lookahead required by the runtime families it currently supports

For the Milestone 3 runtime profile, this field is the contract the pager uses to size seam overlap or validate page-boundary assumptions without reconstructing rule widths from regex source.

### Token kind declaration

```swift
public struct TokenKindDecl: Sendable, Codable, Equatable {
  public let tokenKindID: UInt16
  public let name: String
  public let defaultMode: RuleMode
}
```

This gives Milestone 3 a stable table for output token kinds and debug skip emission.

### Byte class declaration

```swift
public struct ByteClassDecl: Sendable, Codable, Equatable {
  public let classID: UInt8
  public let bytes: [UInt8]   // strictly ascending, unique
}
```

### Class set declaration

```swift
public struct ClassSetDecl: Sendable, Codable, Equatable {
  public let classSetID: UInt16
  public let classes: [UInt8]   // strictly ascending, unique
}
```

The sorted order is part of the artifact contract, not merely an implementation detail.

### Lowered rule

```swift
public struct LoweredRule: Sendable, Codable, Equatable {
  public let ruleID: UInt16
  public let name: String
  public let tokenKindID: UInt16
  public let mode: RuleMode
  public let family: RuleFamily
  public let priorityRank: UInt16      // 0 == highest priority
  public let minWidth: UInt16
  public let maxWidth: UInt16?
  public let firstClassSetID: UInt16
  public let plan: RulePlan
}
```

### Rule ordering

`rules` are stored in source declaration order. That order is the priority order for tie-breaking after longest match.

`priorityRank` should simply make that order explicit to runtime code.

### Keyword remap table

```swift
public struct KeywordRemapTable: Sendable, Codable, Equatable {
  public let baseRuleID: UInt16
  public let baseTokenKindID: UInt16
  public let maxKeywordLength: UInt8
  public let entries: [KeywordRemapEntry]
}

public struct KeywordRemapEntry: Sendable, Codable, Equatable {
  public let lexeme: [UInt8]
  public let tokenKindID: UInt16
}
```

### Artifact invariant

The artifact must be sufficient for Milestone 3 to lex without reinterpreting the original regex source.

That means:

- no runtime regex parser
- no runtime normalization
- no runtime reconstruction of semantic byte predicates
- no ambiguity about family selection
- no need to infer token-kind IDs or keyword tables from rule names
- no jagged transition tables that require semantic reshaping before execution

## Generator diagnostics

The generator itself should emit good diagnostics for spec authors.

### Diagnostic rows

Treat the generator like a compiler component. Give it structured diagnostics:

```swift
public struct SpecDiagnostic: Sendable, Codable, Equatable {
  public let code: SpecDiagCode
  public let severity: Severity
  public let primarySpan: SpecSpan
  public let secondarySpan: SpecSpan?
  public let message: String
}
```

`SpecSpan` now refers to **Swift source locations of the embedded DSL declarations**, not a separate `.lex` file. The implementation may capture those via `#fileID`, `#line`, `#column`, and can be upgraded later to richer spans if the authoring surface adopts macros.

### Examples

- zero-width token rule
- duplicate top-level token kind
- duplicate keyword
- keyword not matched by base token
- local-window width cap exceeded
- fallback state cap exceeded
- rule family not supported by the checked-in runtime profile, if that profile is part of the build check

### Determinism

Diagnostics must be sorted by:

1. file ID
2. start line
3. start column
4. end line
5. end column
6. diagnostic code

The same policy you want later for source diagnostics should already hold here.

## Verification strategy

This milestone should be unusually test-heavy because the generator is pure and therefore cheap to validate.

### Unit tests

- DSL declaration fixtures
- normalization fixtures
- byte-class partition fixtures
- class-set canonicalization fixtures
- family classification fixtures
- fallback DFA construction fixtures
- artifact encoding fixtures

### Property tests

At minimum:

1. **Normalization idempotence**

   ```text
   normalize(normalize(x)) == normalize(x)
   ```

2. **Compile determinism**

   ```text
   compile(spec) bytes == compile(spec) bytes
   ```

3. **Artifact round-trip**

   ```text
   decode(encode(artifact)) == artifact
   ```

4. **No nullable token rule survives validation**

   This should be impossible post-validation.

5. **Byte-class soundness**

   For every two bytes in the same class, primitive predicate membership is identical.

6. **Class-set soundness**

   For every byte set projected into a class set, byte membership and class-set membership agree exactly.

7. **Fallback table layout soundness**

   For every fallback rule, `transitions.count == stateCount * transitionRowStride`, and every referenced successor state is in range.

### Differential tests against a reference matcher

Write a tiny slow interpreter for the normalized regex language and use it only in tests.

For random byte strings up to a modest length, assert that:

- normalized regex matcher
- lowered family plan
- fallback DFA, when applicable

all agree on accepted lengths.

This is exactly the kind of verifiable reward the architecture wants. The generator should not be trusted because the code looks nice; it should be trusted because it survives differential checking.

### Golden tests

Check in:

- the typed Micro-Swift spec source
- the canonical generated artifact JSON
- selected intermediate dumps for normalized regexes, byte classes, and class sets

A spec change should produce a clear artifact diff.

## Suggested module API

A compact API surface could look like this:

```swift
public enum LexerGen {
  public static func declare(_ spec: LexerSpec) -> DeclaredSpec
  public static func normalize(_ spec: DeclaredSpec) throws -> NormalizedSpec
  public static func validate(
    _ spec: NormalizedSpec,
    options: CompileOptions
  ) throws -> ValidatedSpec
  public static func lower(
    _ spec: ValidatedSpec,
    options: CompileOptions
  ) throws -> LexerArtifact

  public static func compile(
    _ spec: LexerSpec,
    options: CompileOptions
  ) throws -> LexerArtifact
}
```

And the compile options:

```swift
public struct CompileOptions: Sendable, Codable, Equatable {
  public let maxLocalWindowBytes: Int
  public let enableFallback: Bool
  public let maxFallbackStatesPerRule: Int
}
```

That API is small, compositional, and easy to test.

CLI, `check`, checked-in artifact, CI integration, and property tests.

This sequencing gives fast feedback. You can approve the spec and intermediate IR before touching fallback logic.

>
> **Goal:** compile a declarative, typed, byte-oriented lexer specification written as an embedded Swift DSL into a deterministic, versioned `LexerArtifact` consumed by the runtime lexer.
>
> **Architecture:** the generator is a pure Swift library with the one-way pipeline
> `Swift LexerSpec value -> declared spec -> normalized regex IR -> validated rule set -> byte classes -> class sets -> family lowering -> canonical artifact`.
> The imperative shell is a thin CLI/build-plugin wrapper only.
>
> **DSL:** author the spec as ordinary Swift values built from a narrow byte-regex DSL. Support literals, byte classes, concatenation, alternation, grouping, and bounded or unbounded repetition. Reject unsupported regex features by construction wherever possible. Keywords are not top-level regex rules; they are deterministic remap tables attached to identifier-like rules.
>
> **Normalization:** desugar regex sugar, flatten concat and union, merge adjacent literals, canonicalize alternation ordering, canonicalize byte classes, compute `nullable`, `minWidth`, `maxWidth`, and `firstByteSet`, and require normalization idempotence.
>
> **Lowering:** every validated rule lowers into exactly one family: `literal`, `run`, `localWindow`, or `fallback`. Classification order is greedy in that order, with `run` preferred over `localWindow` whenever both apply. Residual rules lower to deterministic derivative-based DFAs over byte equivalence classes with stable state numbering and flattened row-major transition tables.
>
> **Artifact:** emit canonical JSON with fixed field order, no timestamps, stable token-kind IDs, `byteToClass[256]`, class membership tables, canonical class-set tables, lowered rule plans, keyword remap tables, and bounded-lookahead runtime hints. >
> **Verification:** DSL declaration goldens, normalization idempotence tests, byte-class and class-set soundness tests, compile-twice-equals tests, artifact round-trip tests, fallback table layout tests, and differential tests against a slow reference matcher. >
> **Acceptance criteria:** the typed Micro-Swift spec compiles successfully; no accepted rule is nullable; token-kind, class-set, rule, and state numbering are deterministic; the canonical artifact is golden-testable and byte-for-byte stable; the checked-in v0 artifact uses only the runtime families implemented by Milestone 3’s initial profile; and all generator checks pass within the CI latency budget.
